# Koradio Agent Operating Guide

> Scope: 本仓库中的所有 AI Coding Agents 与自动化开发工具  
> Purpose: 定义 HOW TO WORK、HOW TO THINK、HOW TO EXECUTE  
> Engineering rules: [AI_RULES.md](AI_RULES.md)

## 1. 基本指令

- 全程使用中文沟通，代码标识符和标准技术名称保留英文。
- 先核实仓库事实，再引用文档结论。
- 明确区分当前实现与目标设计。
- 只处理用户授权的任务范围。
- 发现更优路径时，先说明收益、代价和影响。
- 高影响歧义必须先澄清，不得猜测。
- 简单、低风险、边界明确的改动可直接执行。
- 复杂改动必须先讨论方案并获得确认。

## 2. 强制读取顺序

1. `README.md`：项目阶段、真实目录和可执行状态。
2. `AGENTS.md`：工作流程。
3. `AI_RULES.md`：工程硬约束。
4. `context.md`：稳定领域认知。
5. 与任务直接相关的权威文档。
6. 任务涉及的真实文件、配置、入口和测试。

| Task concern | Required document |
|---|---|
| 产品范围、字段、文案、验收 | `docs/prd.md` |
| 用户路径、状态和异常分支 | `docs/user-flow.md` |
| 系统边界、模块、数据和安全 | `architecture.md` |
| UI、动效和无障碍 | `design/design.md` |
| 高保真原型生成 | `design/prompt.md` 与 `design/references/` |
| 开发前视觉资产制作与验收 | `design/assets/README.md`、`design/tasks/visual-assets.md` 与 `design/design.md` |

不得只依赖摘要文件完成需要精确规则的实现。

## 3. 当前阶段保护

当前项目处于 Documentation-first 阶段。

- 不得声称尚不存在的代码、脚本、端口或依赖已经可用。
- 不得把 `architecture.md` 的目标目录当作真实目录。
- 不得虚构包管理器、版本或运行命令。
- 不得把 `VDA-*` 中的计划产物描述为已经存在的 Figma、HTML、组件或 QA 证据。
- 脚手架出现后必须重新核对 README 的状态描述。

## 4. 标准执行闭环

```text
理解目标 → 核实仓库事实 → 识别歧义与风险 → 规划 → 修改 → 验证 → 汇报
```

### 4.1 理解与核实

- 用一句话确认交付物、成功标准、范围和限制。
- 区分用户要求、仓库事实与 Agent 推断。
- 找出任务涉及的权威文档。
- 读取真实文件树、相关入口、配置和测试。
- 检查已有变更并视为用户资产。
- 不询问可以从仓库直接得到的事实。

### 4.2 澄清

仅在答案会改变交付范围、公共行为、数据或高成本实现时提问。

- 只问当前阻塞决策。
- 给出已发现的仓库事实。
- 给出推荐默认项及其影响。
- 不把可自行验证的问题交给用户。

### 4.3 规划

- 修改超过 3 个文件时，先拆成可独立验证的小任务。
- 复杂改动先说明目标、边界、顺序和验证方法。
- 每个任务必须有独立、可检查的完成条件。
- 计划必须明确创建、修改和保持不变的文件。
- 不把无关重构混入当前任务。
- 开发前视觉资产任务只使用 `VDA-*` 编号，并在 `design/tasks/visual-assets.md` 维护状态和证据。
- 后续正式开发任务使用独立的 `DEV-*` 编号和开发任务文档，不得混入 VDA 清单。
- `context.md` 只记录稳定阶段事实和任务入口，不记录 VDA 动态进度。

### 4.4 修改

- 保持现有命名、格式和文档风格。
- 先做满足目标的最小完整改动。
- 不覆盖、删除或回退用户的无关变更。
- 不执行范围外的外部写入、发布或通知。
- 新发现的问题若不阻断当前任务，记录后继续原任务。

### 4.5 验证

- 运行与风险相匹配的检查。
- 先验证修改文件，再验证受影响边界。
- 检查失败时定位原因，不得删除检查或弱化标准。
- 无法运行检查时，说明缺失条件和未验证范围。
- 没有证据时不得声称完成、通过或修复。

### 4.6 汇报

最终回复先给结果，再给必要证据，并包含：

- 创建或修改了什么。
- 验证了什么以及结果。
- 仍存在什么限制、风险或未决事项。
- 仅在必要时提供用户下一步。

不得复述完整过程、输出无关日志或重复原始需求。

## 5. 文档冲突处理

| Concern | Authority |
|---|---|
| 产品行为 | `docs/prd.md` |
| 用户流程 | `docs/user-flow.md` |
| 系统边界 | `architecture.md` |
| UI 规范 | `design/design.md` |
| 工程规则 | `AI_RULES.md` |
| 工作方式 | `AGENTS.md` |

发生冲突时：

1. 停止实现冲突部分。
2. 指出具体陈述和所属 Concern。
3. 以用户最新明确决策解决冲突。
4. 先修改对应权威文档。
5. 同步受影响的摘要、规则和导航文件。
6. 再继续实现。

不得静默选择对当前实现更方便的版本。

## 6. 失败与阻塞

以下情况必须停止扩大改动：

- 缺少决定公共行为所需的用户选择。
- 权威文档存在无法消解的冲突。
- 必需依赖、凭据或环境不可用。
- 同一验证失败重复出现且无安全替代路径。
- 继续操作会覆盖用户工作或产生范围外副作用。

报告阻塞时说明已完成部分、阻塞条件、已做检查和继续所需的最小输入。

## 7. 完成定义

- [ ] 用户要求的交付物全部存在。
- [ ] 改动没有超出授权范围。
- [ ] 权威文档与实现一致。
- [ ] 适用检查已运行并通过。
- [ ] 未验证部分已明确披露。
- [ ] 无占位符、临时内容或隐藏待办。
- [ ] 用户已有改动未被破坏。
- [ ] 最终回复给出结果和验证证据。

## 8. 最终自检

- 是否把目标设计误写成当前事实？
- 是否遗漏了对应权威文档？
- 是否执行了未授权操作？
- 是否修改了不相关文件？
- 是否用实际检查支持完成声明？
- 下一位 Agent 能否依赖仓库继续工作？
