# Koradio Architecture Decision Records

> Status: Active
> Scope: 会改变工程基线、系统边界、安全边界、工具链或发布路径的高影响决策
> Task registry: [tasks.md](../project-management/tasks.md)

## 1. 作用与权威边界

ADR 记录一项高影响工程决策的上下文、候选方案、裁决、后果和验证证据，确保后续实现能够解释“为什么这样做”。ADR 不替代产品、流程、架构、视觉或工程规则的权威文档，也不能静默覆盖它们。

| Concern | Source of truth |
|---|---|
| 产品行为、字段、文案、验收 | [产品需求文档](../prd.md) |
| 用户路径、状态与异常分支 | [用户流程](../user-flow.md) |
| 系统边界、模块、数据与安全 | [系统架构](../../architecture.md) |
| UI、动效与无障碍 | [视觉设计系统](../../design/design.md) |
| 工程硬约束 | [工程规则](../../AI_RULES.md) |
| 动态任务状态 | [任务登记表](../project-management/tasks.md) |

若 ADR 结论改变上述 Concern，必须在同一任务中先更新对应权威文档，再同步受影响的规则、导航和任务；发生冲突时 ADR 不拥有更高优先级。

## 2. 何时创建 ADR

以下事项必须使用 ADR：

- 工具链、运行版本、依赖管理、测试或构建体系。
- 运行拓扑、端口分配、进程生命周期、会话与 Origin 边界。
- 事实源、模块 owner、公共 contract、数据库归属或依赖方向。
- 数据迁移、秘密存储、文件边界或其他安全模型。
- macOS 包装、签名、公证、升级、回滚或发布路径。
- Provider 接入方式、可行性、授权边界或不可逆替换。

局部实现细节、可随代码安全调整的命名、单次缺陷修复和不改变公共行为的重构通常不创建 ADR。若判断不确定，以“未来更换是否会造成跨模块、高成本或数据/安全风险”为准。

## 3. 编号与文件名

- 文件名使用 `NNNN-short-kebab-case.md`，编号四位、全局递增且永不复用。
- `0000-development-baseline.md` 是 S0-02 的基线记录；后续决策从 `0001` 开始。
- 文件名在创建后保持稳定。标题或状态变化不重命名文件。
- 替代既有决策时创建新编号，并在新旧两份 ADR 中互相链接。
- [template.md](template.md) 不参与编号。

## 4. 状态模型

| 状态 | 含义 | 允许的后续状态 |
|---|---|---|
| `提议` | 候选决策，尚不能作为实现依据 | `已接受`、`已否决` |
| `已接受` | 已裁决并可作为实现依据 | `已取代`、`已弃用` |
| `已否决` | 已评估但不采用 | 无；新信息需要新 ADR |
| `已取代` | 已由另一份 ADR 替代 | 无；必须链接替代者 |
| `已弃用` | 原问题或能力退出范围，不再适用 | 无 |

ADR 只有在决策人、验证证据和受影响权威文档均明确后才能进入 `已接受`。不能用修改旧 ADR 正文的方式抹去历史结论；允许补充状态、替代链接和勘误说明。

## 5. 标准流程

1. 在 `tasks.md` 中确认任务边界、依赖和决策人。
2. 复制 [template.md](template.md)，分配下一个未使用编号并填写为 `提议`。
3. 记录事实、约束、候选方案、裁决标准和验证计划；不把计划描述为当前实现。
4. 对高影响选项取得项目所有者确认，并执行任务要求的 PoC、检查或外部验证。
5. 写入唯一裁决、正负后果、风险、回滚或替代路径，并同步权威文档。
6. 将状态更新为 `已接受` 或 `已否决`，在本索引登记，并通过任务 PR 保存证据。
