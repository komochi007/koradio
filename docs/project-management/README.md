## 0. AI 快速阅读与执行顺序

| 顺序 | 文件 / 信息 | AI 需要得到的答案 | 未满足时的动作 |
|---:|---|---|---|
| 1 | 根 [README](../../README.md) | 当前阶段、真实目录、产品是否可运行 | 以仓库事实修正文档误判，不引用计划目录为当前实现 |
| 2 | [AGENTS.md](../../AGENTS.md)、[AI_RULES.md](../../AI_RULES.md)、[context.md](../../context.md) | 工作方式、工程硬约束、稳定领域认知 | 停止范围外或违反硬约束的动作 |
| 3 | [roadmap.md](roadmap.md) | 当前阶段、阶段门、关键路径和主要风险 | 不绕过未通过的阶段门 |
| 4 | [tasks.md](tasks.md) | 哪个任务 Ready、其依赖/输入/边界/验收是什么 | 只选择状态为 `待开始` 且依赖全部完成的任务 |
| 5 | [git-workflow.md](git-workflow.md) | 如何同步、修改、验证、提交和推送 | 常规任务在根目录 `main` 顺序执行，不创建 worktree 或分支 |
| 6 | 任务“输入”引用的权威文档 | 精确产品、流程、架构或 UI 规则 | 发现冲突时先按 Concern 修正文档 |
| 7 | [release-checklist.md](release-checklist.md) | 当前版本是否满足 Beta、RC 或发布硬门 | 未到 S7 前只阅读，不提前勾选发布项 |

AI 不应一次性加载所有历史视觉任务或全部实现细节。先通过路线图定位阶段，再通过任务表读取当前任务的“输入”字段，按需加载直接相关权威章节和真实文件。

## 1. 目的与权威边界

本目录统一管理 Koradio 的阶段、任务、依赖、交付物和发布门。它回答“接下来做什么、为什么做、完成的证据是什么”，不替代产品、流程、架构、设计或工程规则的权威文档。

| Concern | Source of truth |
|---|---|
| 产品行为、字段、文案、验收 | [产品需求文档](../prd.md) |
| 用户路径、状态与异常分支 | [用户流程](../user-flow.md) |
| 系统边界、模块、数据与安全 | [系统架构](../../architecture.md) |
| UI、动效与无障碍 | [视觉设计系统](../../design/design.md) |
| 工程硬约束 | [工程规则](../../AI_RULES.md) |
| 当前阶段和真实目录 | [项目 README](../../README.md) |
| 动态阶段、任务与发布状态 | 本目录 |

产品或架构事实改变时，必须先更新对应权威文档，再同步任务和路线图。任务清单不得自行定义新的公共行为。

## 2. 管理原则

- 仓库 `main` 上的 Markdown 是任务状态的唯一事实源。本地执行中的 `tasks.md` 随交付物一起提交；GitHub Issue、Project 或临时 PR 只保存外部引用和验证证据，不维护独立状态。
- 不使用无法解释的整体完成百分比；分别汇报产品定义、视觉设计、工程实现、质量验证和发布准备。
- 任务按可独立交付和验证的结果拆分，不按“写一点代码”“继续开发”等活动命名。
- 每个任务只允许一个主要阶段；跨阶段依赖通过任务 ID 明确表达。
- 已完成的 VDA-00～17 属于历史视觉任务，只引用 [视觉交接映射](../../design/assets/reports/handoff-map.md)，不复制为新待办。
- 当前未提交改动均视为用户资产。任何基线整理任务都不得擅自删除、回退、提交或发布现有改动。

## 3. 状态模型

任务状态只使用以下五种值：

| 状态 | 含义 | 进入条件 | 退出条件 |
|---|---|---|---|
| `待开始` | 尚未开始 | 任务已登记 | 负责人开始执行，或确认阻塞 |
| `进行中` | 正在产出交付物 | Ready 条件满足且已有负责人 | 提交验收或确认阻塞 |
| `待验收` | 交付物已完成，等待验证 | 交付物存在且执行者已完成自检 | 验收通过或退回进行中 |
| `已完成` | 验收和证据均成立 | Done 条件全部满足 | 仅在发现完成证据失效时重新打开 |
| `阻塞` | 无法安全继续 | 阻塞条件真实发生且无范围内替代路径 | 阻塞解除后回到待开始或进行中 |

推荐流转：

```text
待开始 → 进行中 → 待验收 → 已完成
             ↘ 阻塞 ↗
```

## 4. 任务字段

每个任务必须包含以下字段：

| 字段 | 规则 |
|---|---|
| ID | 全局唯一，使用 `S{阶段}-{两位序号}` |
| 阶段 | `S0` 至 `S9` 之一 |
| 状态 | 只使用本规范的五种状态 |
| 优先级 | `Critical`、`High`、`Medium`、`Low`；不复用 PRD 的 P0/P1 产品优先级 |
| 目的 | 一至两句直白说明要解决的问题和用户或工程价值 |
| 依赖 | 完成前必须满足的任务 ID 或外部条件；没有则写“无” |
| 输入 | 执行任务必须读取或取得的权威资料、决策和凭据 |
| 范围边界 | 明确本任务做什么以及不做什么 |
| 验收标准 | 可观察、可重复检查的完成条件 |
| 交付物 | 代码、配置、报告、安装包或记录 |
| 生成路径 | 预期创建或修改的目录和文件；未创建前必须标注为计划路径 |
| 验证方式 | 具体检查、测试、人工验收或外部验证 |
| 负责人 | 未分配时写 `待分配`，不得用 Agent 名代替责任人 |
| 外部引用 | 可选 GitHub Issue、PR、ADR 或发布记录；没有则写“无” |
| 阻塞条件 | 会阻止任务继续的最小条件及解除要求 |

## 5. Ready 与 Done

任务进入 `进行中` 前必须满足 Definition of Ready：

- 目的、边界、依赖、验收标准和交付物明确。
- 所有依赖已完成，或任务明确允许并行且不会猜测公共行为。
- 相关权威文档无未解决冲突。
- 必需环境、凭据、测试替身或用户决策可用。
- 不会覆盖未提交的用户工作。

任务进入 `已完成` 前必须满足 Definition of Done：

- 交付物存在，且没有占位内容或隐藏 TODO。
- 适用测试、检查和人工验收已执行并通过。
- 新增路径、命令和运行行为已同步到 README 或对应权威文档。
- 失败和未验证范围已披露；不得通过弱化测试完成任务。
- `tasks.md` 已记录完成证据或外部引用。

## 6. 优先级与执行顺序

| 优先级 | 定义 |
|---|---|
| `Critical` | 阻断阶段门、数据安全、发布或核心闭环；必须先处理 |
| `High` | 阶段核心交付，延迟会影响关键路径 |
| `Medium` | 非阻断质量、效率或完整度工作 |
| `Low` | 可在不影响 v1.0 验收时延后 |

实际顺序由“已满足的依赖 → 更高优先级 → 关键路径”共同决定。优先级不能绕过依赖和权威文档冲突。

## 7. 阶段门

- 阶段任务可以并行，但下一阶段的关键路径任务只有在上一阶段门通过后才能开始。
- 阶段门是验收结果，不是日历日期；具体条件见 [roadmap.md](roadmap.md)。
- 首轮外部 Beta 只有在 S5、S6、S7 全部通过后才能开始。
- 阶段门未通过时，不得通过把任务移到后续阶段来隐藏缺口。

## 8. 阻塞与变更

任务标记为 `阻塞` 时必须记录：已完成部分、阻塞事实、已执行检查、解除阻塞所需的最小输入。依赖未排期、任务困难或预计耗时长本身不构成阻塞。

发生以下变化时必须执行变更控制：

- 产品行为变化：先更新 `docs/prd.md` 和受影响用户流程。
- 模块 owner、公共 contract、数据库归属或事实源变化：先更新 `architecture.md`。
- UI token、页面骨架、组件状态或无障碍要求变化：先更新 `design/design.md` 和视觉主源。
- 工具链、发布形态或安全边界变化：新增或更新 ADR，并同步 README、任务依赖和路线图。

## 9. 日常更新流程

1. 从 `tasks.md` 选择依赖已满足的任务。
2. 核对权威文档、真实文件树、当前 Git 状态和已有变更。
3. 按 [Git 工作流](git-workflow.md) 在根目录同步 clean `main`；确认没有其他任务或 Agent 同时修改仓库。
4. 本地更新任务为 `进行中`，完成交付物并执行任务定义的验证，无需单独提交状态变更。
5. 验收通过后更新为 `已完成`并同步状态摘要；若该任务关闭阶段门，同步 `roadmap.md`。
6. 显式暂存本任务文件，提交并直接推送 `origin/main`；仅在用户要求或确认高风险隔离时使用临时分支和 PR。
7. 发布阶段同时更新 [release-checklist.md](release-checklist.md)，不得只改任务状态。

## 10. 任务表新增行模板

新增任务必须放入 [tasks.md](tasks.md) 对应阶段表，不得恢复成散落的标题或复选框任务。使用以下列顺序：

| ID / 任务 | 状态 | 优先级 | 目的 | 依赖 | 输入 | 范围边界 | 验收标准 | 交付物 / 计划路径 | 验证方式 | 负责人 / 外部引用 | 阻塞条件 |
|---|---|---|---|---|---|---|---|---|---|---|---|
| `S0-00` 可验收任务名 | 待开始 | High | 一至两句说明结果和价值 | 无或任务 ID | 权威文档、ADR、环境或凭据 | 明确包含与不包含内容 | 可观察、可重复检查的 Done 条件 | 计划文件、代码、报告或产物路径 | 实际命令、测试或人工验收 | 待分配 / 无 | 最小阻塞事实；解除条件 |
