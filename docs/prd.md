# Koradio 产品需求文档

## 一、概述

**文档信息**

| 版本号 | 日期 | 作者 |
|--------|------|------|
| V 1.1.0（开发前校准） | 2026-07-15 | 用户 / Codex |

### 1. 需求背景

- **目标用户现状**：目标用户是有长期听歌习惯、收藏歌单和明确音乐品味的音乐重度用户。他们会根据工作、写作、通勤、夜晚放松等场景主动找歌，但现有音乐产品多以算法流、榜单、歌单广场为主，难以像私人 DJ 一样理解当下语境并完成策展式播放。
- **核心痛点**：用户想要“说一句场景需求，就得到一段有开场、有串讲、有歌单、有播放反馈的私人 AI 电台”，而不是自己反复搜索、切歌、拼歌单。
- **当前问题汇总**：

| 问题点 | 问题描述 | 当前影响 |
|--------|----------|----------|
| 场景理解弱 | 普通播放器只知道歌曲和歌单，不理解用户此刻在做什么、想要什么氛围 | 用户需要手动搜索、试错和切歌 |
| 推荐解释弱 | 算法推荐通常不给出有品味、有情境感的推荐理由 | 用户感觉像被动接收列表，而不是被私人 DJ 策展 |
| 电台体验断裂 | 播放器、歌词、AI 文案、语音串讲通常分散在不同产品或功能里 | 无法形成“开场-播放-转场-反馈”的完整电台闭环 |
| 个人品味沉淀弱 | 用户的歌单、反馈、场景偏好没有被整理成可读、可编辑的个人品味档案 | 产品难以越用越懂用户 |

### 2. 需求目标

- **业务目标**：
  - 以“场景输入到电台播放”为 MVP 核心体验；`2–3 周` 仅是脚手架确定后的非承诺性估算，不作为发布日期或验收条件。
  - 支持本地多档案使用，让不同内测用户能在同一台机器上拥有独立品味和历史；设备级服务配置不随档案复制。
  - 建立可扩展的本地服务架构，为后续 24/7 自动电台、日程联动和更多音乐源预留空间。

- **成功标准**：

| 指标 | 当前状态 | 目标状态 |
|------|----------|----------|
| 场景点歌闭环 | 当前无产品实现 | 在 Codex 与网易云可用时，提交合法场景后生成至少 1 首可播放歌曲和 1 段开场文字；新节目提交成功后原子切换并开始播放 |
| 首次启动 | 当前无配置流程 | 首次启动自动选择可写的 OS 应用数据目录；用户只需创建档案并完成 Codex 配置，内置网易云 Provider 通过健康检查后即可发起首次生成 |
| DJ 串讲降级 | 当前无串讲 | TTS 缺失、无效或超时时仍保存文字 DJ，时间线不创建伪音频项，歌曲可继续播放 |
| 推荐反馈沉淀 | 当前无反馈记录 | 喜欢、取消喜欢、不喜欢、取消不喜欢、节目收藏、取消收藏和跳过均追加显式事件，重建自动品味时不覆盖人工规则 |
| 异常可恢复性 | 当前无错误处理 | 完全离线、生成失败、Profile 切换、数据迁移失败和多标签接管均有确定结果，不破坏既有节目、旧数据或人工设置 |
| 视觉基线一致性 | VDA-17 已冻结 | VDA-17 HTML 主源与 60 张 `960 × 1600px` 正式基线持续通过复渲染校验；语义修正不产生像素差异 |

### 3. 需求范围

- **包含**：
  - 本地 Web/PWA 竖版电台界面。
  - Node.js 本地服务中枢。
  - 本地 Codex 编排：读取用户档案、生成节目计划、输出结构化播放指令。
  - 本地服务内置的网易云 Provider：通过 TypeScript `linuxapi` 适配器完成音乐搜索、歌曲信息、播放链接和歌词获取，v1 不要求用户填写网易云地址或密钥。
  - 可选 Apple 系统 TTS：由 bundled macOS native helper 调用 `AVSpeechSynthesizer` 和当前设备已安装的标准系统语音；不可用时降级为文字 DJ，v1 不使用云 TTS 或 Personal Voice。
  - 本地多档案：用户档案、品味标签、播放历史和反馈；Codex 与数据目录属于设备级配置，网易云与 TTS 属于设备级运行时能力。
  - Radio 主播放页：顶部品牌区、时间/ON AIR、播放器、队列、DJ 状态栏、DJ 对话区、底部输入的单列固定顺序。
  - DJ 状态栏详情态：全屏沉浸节目界面、深色声波区、浅色节目面、AI 串讲词、歌词、歌曲进度和节目整体进度。
  - 昼夜模式：夜间和日间都按冷调高级电台风格完整设计。
  - AI DJ 人格与语言：英式电台主持人气质，支持中文或英文串讲。
  - Library、Taste、Programs/History、Settings 辅助页面。

- **不包含**：
  - 云同步、跨设备同步和云端账号系统。
  - 商业支付、订阅、会员权益。
  - 公开社区、歌单广场、多人同时收听。
  - 完整 24/7 自动电台和复杂日程自动编排。
  - 多音乐源聚合，第一版仅以网易云 API 为 MVP 音乐源。
  - 真实音频波形预分析，MVP 可使用前端模拟动态柱状图。

### 5. 产品体验与视觉原则

**体验气质**

Koradio 的第一版体验应是“私人电台陪伴感 + 简约高级风格”。产品需要像一个克制、懂音乐、不会过度打扰的私人 DJ，而不是工具型音乐控制台。界面保留点阵字体、`ON AIR`、状态灯、波形、队列、`LIVE` 状态等电台识别元素，但避免复杂装饰、过度拟物和大面积花哨渐变。

UI 品牌字标使用全大写 `KORADIO`，字距略微拉开；正文叙述和文档中仍使用产品名 Koradio。Logo 应采用单色几何低细节设计，避免直接使用麦克风、耳机、唱片等常见音乐应用符号。

**昼夜模式**

| 模式 | 视觉基准 | 使用规则 |
|------|----------|----------|
| Night | 深黑、墨蓝、低亮度边框、绿色状态光、白色点阵数字 | 默认首选模式，突出深夜私人电台感 |
| Day | 冷白、浅灰、黑色正文、绿色状态光、低饱和分割线 | 与 Night 使用同一布局和组件结构，不做简单反色 |

用户可在 Settings 中为当前 Profile 选择 `Dark`、`Light` 或 `System`。MVP 不根据歌曲封面自动变色；当前歌曲可影响小范围状态光或波形强弱，但不得改变整体主题基准。

**语言规则**

Radio 主播放页和 Radio Detail Sheet 尽量使用英文状态词，例如 `ON AIR`、`LIVE`、`QUEUE`、`PLAYING`、`SPEAKING`、`THINKING`、`Now Playing`、`Say something to the DJ...`。Onboarding/Profile、Library、Taste、Programs/History、Settings 以中文为主，降低配置和管理成本。DJ 串讲语言由当前 Profile 的 `DJ Language` 决定，支持中文或英文，不做全应用完整国际化。

**AI DJ 人格与文案规则**

| 维度 | 规则 |
|------|------|
| 人格 | 英式电台主持人气质，声音柔和有磁性，专业克制，带轻微冷幽默 |
| 说话时机 | 每次节目必须有开场；只在情绪切换、风格变化、用户追问或关键歌曲前后做转场 |
| 说话长度 | 开场 20-45 秒；转场 8-20 秒；结束语 10-25 秒 |
| 内容范围 | 可简单介绍歌曲背景、风格特点、年代、场景关联或推荐理由 |
| 禁止内容 | 不油腻、不煽情过度、不装腔、不长篇百科、不每首歌都打断、不编造无法确认的音乐事实 |
| 失败降级 | 无法确认歌曲背景时，只描述可感知的氛围和用户场景，不输出虚构资料 |

**设计落地约束**

| 维度 | 规则 |
|------|------|
| 原型画布 | 高保真原型以 `960 × 1600px`、`3:5` 竖版桌面应用截图为基准，不展示设备模型、浏览器外框或营销落地页 |
| 桌面 PWA 画布 | 精细指针桌面设备在窗口宽度达到 `960px` 时使用居中的 `960 × 1600px` 固定产品画布，不自动缩放、不显示页面级滚动条；窗口高度不足 `1600px` 时只显示尺寸提示，达到要求后自动恢复当前页面 |
| 内容列宽 | Radio `816px`；服务异常恢复区 `560px`；Profile/Onboarding `848px`；Library/Taste/Programs `840px`；Settings `832px`；Detail Sheet 覆盖完整 `960 × 1600px` 画布，浅色节目面内使用 `848px` 内容列 |
| 原型坐标 | Radio 04/05/06 使用 VDA-17 固定骨架：Radio rail `x 72 / w 816`，输入框 `x 72 / y 1372 / w 816 / h 88`，Tab Bar `x 170 / y 1480 / w 620 / h 88` |
| 安全边距 | 原型页面左右使用 `56px`、顶部使用 `40px` 安全边距；底部预留固定导航和安全区 |
| 底部导航 | 一级页面使用固定底部胶囊式 Tab Bar，`620 × 88px`，5 个 `124 × 88px` tab slot，选中底 `88 × 64px`；提供 Tooltip、`aria-label` 和键盘 Focus |
| 按钮颜色 | 主按钮使用黑白高对比，不使用绿色填充；绿色只用于 `ON AIR`、`LIVE`、`PLAYING`、`CONNECTED`、Focus Ring、细线和少量波形 |
| 字体 | 正文使用系统字体；等宽字体仅用于时间、数字、状态标签和队列序号；正文和 DJ 串讲不得使用点阵字体 |
| 页面结构 | Radio 始终保持中央窄列，不做桌面多栏；管理页保持单列，不做后台式侧边栏 |
| 滚动边界 | 固定桌面画布的页面外层、一级内容区和产品外壳不得滚动或展示滚动条；长歌词、DJ 串讲、列表、诊断与表单条目只能在固定的内部内容区域滚动，内部滚动条始终隐藏 |
| DJ 对话区 | Radio 主页面使用 `816 × 288px` 独立 dialogue well，保留弱背景和上下边界；DJ 文案用自然段，用户输入用右对齐弱气泡 |
| Detail Sheet | 覆盖完整产品界面，不露出底层 Radio 页面；使用深色声波区、纯白节目面 `#FFFFFF` 和浅灰串讲词/歌词卡片 `#F5F3F6`；不显示专辑封面、上一首、下一首、喜欢按钮或更多按钮 |
| 卡片使用 | 不把所有区块都做成悬浮卡片；Radio 中仅播放器、队列、输入区等需要边界的模块卡片化 |
| 错误提示 | 非阻断反馈可用 Toast；阻断错误必须在相关区块内联显示，并提供重试、跳转 Settings、修改输入或降级继续播放入口 |
| 动效 | 动效只表达声音、状态和空间关系；避免持续装饰动画、大幅弹跳、高频闪烁和横向页面滑动 |
| 无障碍 | 所有正式组件支持键盘操作、可见 Focus Ring、WCAG AA 对比度、`aria-live` 重要状态播报、200% 浏览器缩放和 `prefers-reduced-motion` |

## 二、页面结构

### 1. 导航结构

| 导航层级 | 页面名称 | 页面定位 | 入口 | 主要承载功能 |
|----------|----------|----------|------|--------------|
| 异常态 | 本地服务未连接 | 本地服务不可用时的恢复入口 | 打开产品且服务未响应 | 重连、只读 Settings、查看脱敏诊断与启动说明 |
| 一级 | Onboarding/Profile | 首次进入、档案选择和档案创建 | 首次打开、顶部头像、登录/档案入口 | 创建本地档案、选择档案、基础偏好录入 |
| 一级 | Radio 主播放页 | 核心电台体验入口 | 默认打开、底部胶囊导航 Radio | 场景输入、DJ 串讲、播放控制、队列、反馈 |
| 一级 | Library | 音乐库和歌单来源管理 | 底部胶囊导航 Library、Radio 队列空状态入口 | 搜索歌曲、导入歌单、查看候选歌曲 |
| 一级 | Taste | 个人音乐品味档案 | 底部胶囊导航 Taste、Profile 完成后入口 | 查看和编辑风格标签、喜欢/避雷规则、场景偏好 |
| 一级 | Programs/History | 节目单与播放历史 | 底部胶囊导航 Programs、Radio 播放完成入口 | 查看历史节目、重播串讲、复用场景 |
| 一级 | Settings | 本地服务、设备配置、数据目录与 Profile 偏好 | 底部胶囊导航 Settings、异常提示入口 | Codex 命令路径、内置网易云 Provider 健康状态、可选 TTS、受控数据目录迁移、主题和 DJ 偏好 |
| 二级详情态 | Radio Detail Sheet | 当前节目沉浸详情 | 用户点击 Radio 页 DJ 状态栏 | 波形、AI 串讲词、歌词、歌曲进度、节目整体进度、单一播放/暂停 |

### 2. 页面职责说明

| 页面名称 | 用户目标 | 关键内容/组件 | 空状态 | 异常状态 |
|----------|----------|----------------|--------|----------|
| 本地服务未连接 | 恢复本地服务连接 | `○ OFFLINE`、断开波纹状态图标、重新连接、只读 Settings 入口、`LOCAL SERVICE · NOT RESPONDING` | 本页面为空状态不适用 | 仅已打开或已缓存的 PWA 可进入只读 Settings；所有配置、测试与迁移控件禁用，只显示启动说明、脱敏状态和重试 |
| Onboarding/Profile | 创建或选择本地电台档案 | 头像、昵称、电台名、常听风格、默认场景、档案列表 | “还没有电台档案，创建你的私人电台” | 本地数据目录不可写时提示检查权限 |
| Radio 主播放页 | 输入场景并完成电台播放 | 单列固定顺序：顶部品牌区、时间/ON AIR、播放器、队列、DJ 状态栏、DJ 对话区、底部输入；主状态词使用英文 | `Say something to the DJ...` | 本地服务断开时显示“Koradio 服务未连接” |
| Library | 准备音乐候选来源 | 搜索框、歌单导入、歌曲列表、加入候选池按钮 | “还没有导入歌单，可先搜索一首歌试播” | 网易云 API 不可用时提示切换关键词或重试 |
| Taste | 管理个人音乐品味 | 品味标签、避雷规则、场景规则、反馈摘要、编辑按钮 | “播放和反馈后会在这里形成你的音乐品味” | 档案读取失败时提示重新选择档案 |
| Programs/History | 回看和复用历史节目 | 节目卡片、播放记录、DJ 文案、重播按钮、复用场景按钮 | “还没有节目，去 Radio 生成第一段电台” | 历史音频文件缺失时保留文本并提示无法重播音频 |
| Settings | 配置本地运行依赖和当前 Profile 偏好 | 本地服务状态、Codex 路径、内置网易云 Provider 状态、Apple 系统 TTS 状态、数据目录、主题模式、DJ 语言、DJ 声音风格、测试与迁移入口 | “完成 Codex 配置并确认音乐服务可用后即可开始播放” | Provider 或原生 helper 超时、迁移或配置保存失败时显示具体原因；完全离线时只读 |
| Radio Detail Sheet | 沉浸查看当前节目 | 全屏沉浸节目界面、顶部状态、全宽动态波形、纯白节目面 `#FFFFFF`、节目标题、简洁当前歌曲、歌曲进度、`#F5F3F6` 串讲词/歌词卡片、节目整体进度、关闭按钮；DJ 段串讲为主，歌曲段歌词为主 | `No live session yet` | 歌词缺失、TTS 失败、播放失败时显示降级内容 |

### 3. Radio 主播放页布局规则

Radio 主播放页采用单列固定顺序，不因主题模式变化而改变信息架构。首屏优先建立电台仪式感，完整露出顶部品牌区、时间/日期、电台状态和播放器；队列、DJ 状态栏、DJ 对话区和底部输入从下方自然延续。空节目状态显示 `LIVE`，节目准备完成或正在播放时显示 `ON AIR`，生成中显示 `TUNING`。

高保真生图原型中的 04/05/06 三态必须共用同一套固定骨架：

| 固定区域 | 原型坐标与尺寸 | 说明 |
|----------|----------------|------|
| 顶部品牌工具区 | `x 56 / y 40 / w 848 / h 64` | Logo、`KORADIO` 字标、头像、主题按钮 |
| Radio 内容 rail | `x 72 / w 816` | 时间、播放器/空态/生成态、队列、DJ 状态栏、DJ 对话区、输入框 |
| 时间状态区 | `x 72 / y 128 / w 816 / h 148` | 时间、日期、`LIVE/ON AIR/TUNING/OFFLINE` |
| 状态主内容区 | `x 72 / y 292 / w 816 / h 340` | 空状态、播放器或生成中核心内容 |
| 队列区 | `x 72 / y 656 / w 816 / h 300` | 空队列、真实队列或骨架队列 |
| DJ 状态栏 | `x 72 / y 980 / w 816 / h 64` | 点击打开 Detail Sheet |
| DJ 对话区 | `x 72 / y 1068 / w 816 / h 288` | 独立 dialogue well，DJ 自然段和用户弱气泡 |
| 底部输入 | `x 72 / y 1372 / w 816 / h 88` | 输入、语音和发送 |
| 底部导航 | `x 170 / y 1480 / w 620 / h 88` | 5 个等宽图标 Tab |

## 四、功能说明

### 功能 1：本地档案创建与选择

**功能描述**

用户可在 Onboarding/Profile 页面创建或选择本地电台档案，隔离各 Profile 的品味、历史与展示偏好；设备级服务配置在所有 Profile 间共享。

**用户操作路径、界面元素、交互逻辑、数据变化**

- 用户点击“创建电台档案” → 系统打开档案表单 → 用户看到头像、电台名、昵称、常听风格、默认场景输入项。
- 用户点击“保存并进入 Koradio” → 系统创建 `Profile`、默认 `TasteOverrides` 与 `ProfilePreferences` → 用户看到 Radio 主播放页进入空播放状态。
- 用户点击已有档案卡片 → 系统加载该档案的 `EffectiveTaste`、节目历史和 `ProfilePreferences`，并读取共享 `DeviceSettings` → 用户看到对应档案的 Radio 页和头像。
- 正在生成或播放时切换 Profile → 取消旧 Profile 的生成任务并丢弃其迟到事件，保存旧播放 checkpoint、停止旧时间线，再加载新 Profile。
- 数据变化：新增或更新 `Profile`、`TasteOverrides`、`ProfilePreferences`；当前 Profile ID 只写入本机 bootstrap 运行配置。

**用户流程**

```text
[首次打开 Koradio]
  ↓
[检查是否存在本地档案]
  ├─ 无档案 → 用户填写档案表单
  │          ├─ 保存成功 → 创建 Profile + TasteOverrides + ProfilePreferences → 进入 Radio
  │          └─ 保存失败 → 提示本地目录不可写或字段错误
  └─ 有档案 → 展示档案列表
             ├─ 用户选择档案 → 加载档案 → 进入 Radio
             └─ 加载失败 → 提示重新选择或检查本地数据
```

**状态机**

| 状态名称 | 进入条件 | 退出条件 | 用户可见表现 |
|----------|----------|----------|--------------|
| 未初始化 | 首次打开且无本地档案 | 用户创建档案成功 | 显示创建档案引导 |
| 选择档案 | 检测到已有本地档案 | 用户选择档案成功 | 显示档案卡片列表 |
| 编辑中 | 用户点击创建或编辑 | 保存成功或取消 | 显示档案表单 |
| 加载中 | 用户选择档案 | 加载成功或失败 | 按钮显示“正在进入电台...” |
| 已进入 | 档案加载成功 | 用户切换档案或退出 | 进入 Radio 主播放页 |
| 错误 | 本地目录不可写或档案损坏 | 用户重试或更换路径 | 显示错误提示和重试入口 |

**字段规范**

| 默认场景 | 文本 | 否 | 120 个字符 | 可为空 | “默认场景不能超过 120 个字符” |

| 档案损坏 | profile JSON/DB 读取失败 | 跳过损坏档案，允许重新选择 | “该档案无法读取，可切换档案或检查本地文件” |

### 功能 2：场景点歌电台生成

**功能描述**

用户可在 Radio 主播放页底部输入框描述当前场景，系统通过本地 Codex 编排生成节目计划、DJ 串讲、候选歌曲和播放队列，解决用户手动找歌和拼歌单的问题。

**用户操作路径、界面元素、交互逻辑、数据变化**

- 用户在输入框输入“今晚写东西，想要安静但不死板的 BGM” → 系统校验输入并锁定发送按钮 → 用户看到 DJ 状态栏变为 `THINKING`。
- 用户点击发送按钮 → 系统读取 `EffectiveTaste`、节目历史、当前时间、`ProfilePreferences` 和共享 `DeviceSettings`，请求本地 Codex 输出节目计划 → 用户看到 DJ 对话区出现 `Tuning your station...`。
- Codex 生成串讲 → 系统按英式电台主持人规则控制语气、长度和说话频率 → 用户看到一段克制、有场景感的开场，而不是每首歌前都被打断。
- Codex 返回计划 → 系统调用网易云 API 搜索歌曲、获取播放链接和歌词 → 用户看到队列逐步生成。
- TTS 可用且生成成功 → 系统将带音频的 `dj` 项插入 `PlaybackTimeline` → 用户听到 DJ 开场并看到串讲文本；TTS 缺失或失败时，文字串讲只保留在 Program segment 中，不创建伪音频项。
- 当前已有节目时，旧节目在新节目生成期间继续播放；生成失败时保持旧节目不变；新节目完整提交后保存旧 checkpoint、停止旧时间线并原子切换。
- 数据变化：新增 `Program`、`DjScriptSegment`、`MusicTrack` 与已提交的 `PlaybackTimeline`；场景输入在提交前只属于前端 draft，不写入持久模型。

**用户流程**

```text
[Radio 底部输入框]
  ↓
[用户输入场景并发送]
  ├─ 输入为空 → 提示输入场景，不发起请求
  └─ 输入有效 → 构建上下文
       ↓
     [请求本地 Codex 生成节目计划]
       ├─ 成功 → 搜索网易云歌曲
       │          ├─ 有歌曲 → 获取播放链接/歌词 → 可选生成 TTS → 原子提交新节目 → 开始播放
       │          └─ 无歌曲 → 换关键词重试 2 次 → 仍失败则提示换个描述
       ├─ Codex 失败 → 保留用户输入并提示重试
       └─ 超时 60 秒 → 中断规划并提示重试
```

**状态机**

| 状态名称 | 进入条件 | 退出条件 | 用户可见表现 |
|----------|----------|----------|--------------|
| 空闲 | Radio 已连接且无生成任务 | 用户发送场景 | 输入框可编辑，DJ 状态为 `LIVE` |
| 校验失败 | 输入为空或超过限制 | 用户修改输入 | 输入框下方显示错误 |
| 规划中 | 用户发送有效场景 | Codex 返回、失败或超时 | DJ 状态为 `THINKING`，发送按钮禁用 |
| 搜歌中 | Codex 返回候选策略 | 搜歌成功或失败 | 队列区显示骨架项 |
| 合成中 | 歌曲候选可用且有 DJ 文案 | TTS 成功或失败 | DJ 状态保持 `THINKING`；TTS 段实际播放时 Detail Sheet 显示 `SPEAKING` |
| 可提交 | 队列和至少一首歌曲可用 | 原子提交成功或失败 | 旧节目继续播放；提交成功后一次性切换 |
| 播放中 | 音频开始播放 | 暂停、结束或失败 | 播放按钮切换为暂停 |
| 失败 | Codex、搜歌或播放失败 | 用户重试或修改输入 | 显示错误提示和重试入口 |

**字段规范**

| 字段名 | 类型 | 是否必填 | 长度限制 | 校验规则 | 错误提示 |
|--------|------|----------|----------|----------|----------|
| 场景输入 | 文本 | 是 | 1-500 个字符 | 去除首尾空格后不可为空 | “告诉 DJ 你现在想听什么” |
| 语音输入转写文本 | 文本 | 否 | 500 个字符 | 转写失败时不覆盖原输入 | “没有听清，可以再说一次” |

**文案规范**

| 场景 | 文案内容 |
|------|----------|
| 空状态 | Say something to the DJ... |
| 加载中 | Tuning your station... |
| 操作成功 | Your session is on air |
| 操作失败 | 这次没有规划成功，请稍后重试 |
| 权限不足 | 请先在 Settings 配置 Codex |
| 网络异常 | 音乐服务连接失败，请检查网络或稍后重试 |

**异常情况处理**

| 异常场景 | 触发条件 | 处理方式 | 用户反馈 |
|----------|----------|----------|----------|
| Codex 失败 | Codex 返回错误或非结构化结果 | 保留用户输入和上下文，允许重试 | “Codex 没有完成节目规划，点击重试” |
| Codex 超时 | 60 秒未返回 | 中断请求，释放输入框 | “规划时间太久了，请重试或缩短描述” |
| 网易云无结果 | 搜索候选为空 | 自动换关键词重试 2 次 | “没有找到合适歌曲，换个说法试试” |
| TTS 失败 | TTS 45 秒超时或返回错误 | 保留文字 DJ，跳过语音串讲 | “语音合成失败，本次先用文字 DJ” |
| 播放链接失效 | 歌曲 URL 不可播放 | 跳过该曲并尝试下一首 | “这首暂时不能播放，已切到下一首” |

| Apple 系统 TTS 不可用 | 原生 helper 不可用、没有匹配语言的标准系统语音或能力检查失败 | 直接使用文字 DJ，不阻止生成 | “系统语音不可用，本次使用文字 DJ” |

### 功能 3：播放控制与队列管理

**功能描述**

用户可在 Radio 主播放页控制当前节目播放，查看和管理队列，解决生成节目后无法主动调整播放节奏的问题。

**用户操作路径、界面元素、交互逻辑、数据变化**

- 用户点击播放/暂停按钮 → 系统暂停或恢复当前 TTS/歌曲音频 → 用户看到按钮状态、进度条和 DJ 状态同步变化。
- 用户点击下一首 → 系统结束当前音频并切换到队列下一项 → 用户看到当前歌曲、进度条、歌词和队列高亮更新。
- 用户点击心形按钮 → 系统追加“喜欢歌曲”事件；用户通过 More 标记不喜欢；节目收藏只在 Programs/节目入口处理。
- 系统生成节目成功 → 系统默认展开队列中的前 3-5 首歌曲 → 用户看到当前节目由哪些歌组成，并能确认当前高亮曲目。
- 用户点击 `HIDE/LIST` → 系统折叠或展开队列区 → 用户看到队列列表隐藏或显示。
- 用户拖动进度条 → 系统 seek 到目标时间 → 用户看到串讲高亮或歌词高亮同步跳转。
- 运行时变化：当前 index、音量、进度、队列展开态、Detail 开关和高亮行都由 Browser Audio Engine 或页面 UI state 管理。仅在切歌、暂停、Profile 切换、标签接管和节目切换等边界保存带当前 `leaseEpoch` 的低频 `PlaybackCheckpoint`；服务端拒绝低于已保存 epoch 的旧标签页写入，跳过通过 `track_skipped` 事件记录。
- checkpoint 的 `positionMs` 不得超过目标 timeline item 时长；只有位于最后一个 item 且精确到达该 item 末尾时才能保存 `completed`，并在同一事务中把 Program 标记为 `completed`。

**用户流程**

```text
[Radio 播放器区域]
  ↓
[用户操作播放控件]
  ├─ 播放/暂停 → 更新 Browser Audio Engine → UI 同步状态
  ├─ 下一首/上一首 → 切换 queue index → 加载对应音频
  ├─ 拖动进度 → seek 音频 → 同步串讲/歌词高亮
  ├─ 生成后队列 → 默认展开 3-5 首
  ├─ 折叠队列 → 仅更新当前标签页 UI state
  └─ 播放失败 → 尝试下一首或提示重试
```

**状态机**

| 状态名称 | 进入条件 | 退出条件 | 用户可见表现 |
|----------|----------|----------|--------------|
| 无队列 | 尚未生成节目 | 生成节目成功 | 只显示 `QUEUE` 状态栏和 `0 TRACKS` |
| 队列展开 | 节目生成成功 | 用户点击 `HIDE` | 默认展示前 3-5 首，当前曲目高亮 |

| 暂停 | 用户点击暂停 | 用户点击播放或切歌 | 按钮显示播放，进度停止 |

### 功能 5：反馈与品味沉淀

**功能描述**

用户可对歌曲执行喜欢、不喜欢、撤销与跳过，也可在 Programs/节目入口收藏或取消收藏节目。系统以 append-only 事件保存每次意图，并据此重建自动品味投影。全屏 Detail Sheet 只承载沉浸阅读和播放/暂停，不放置反馈控件。

**用户操作路径、界面元素、交互逻辑、数据变化**

- 用户点击 Radio 心形按钮 → 系统追加 `track_liked`；再次取消时追加 `track_like_removed`，不覆盖旧事件。
- 用户从 More 选择不喜欢 → 系统追加 `track_disliked`；撤销时追加 `track_dislike_removed`。
- 用户跳过歌曲 → Browser Audio Engine 切到下一项并追加 `track_skipped`。
- 用户在 Programs/节目入口收藏或取消收藏 → 分别追加 `program_favorited` 或 `program_favorite_removed`。
- 数据变化：只新增 `FeedbackEvent`；`TasteProjection` 和界面反馈态由事件流重建，人工 `TasteOverrides` 不被覆盖。
- 自动投影按数据库分配的稳定追加顺序回放每个 Profile 的完整事件流：歌曲喜欢/不喜欢与节目收藏分别按对应事件对的最新有效状态计算，removed 事件只撤销对应状态；`track_skipped` 保留为可追溯事实并推进 projection 版本，但 v1 不把单次跳过推断为不喜欢。
- v1 不从反馈目标猜测风格标签；自动 `tags` 保持为空。有效歌曲喜欢生成 `track:<targetId>` affinity，有效歌曲不喜欢生成同格式 avoid signal，有效节目收藏生成 `program:<targetId>` affinity。
- `sourceVersion` 等于该 Profile 已成功持久化的去重事件数；重复 idempotency key 返回原事件，不推进版本。重建使用全部事件重新计算同一结果，不修改 `FeedbackEvent` 或 `TasteOverrides`。

**用户流程**

```text
[Radio、More 或 Programs]
  ↓
[用户触发反馈]
  ├─ 喜欢/撤销 → 追加 track_liked / track_like_removed → 更新投影
  ├─ 不喜欢/撤销 → 追加 track_disliked / track_dislike_removed → 更新投影
  ├─ 跳过 → 切到下一首 → 追加 track_skipped
  └─ 收藏/取消收藏节目 → 追加 program_favorited / program_favorite_removed
       └─ 保存失败 → 播放继续，提示反馈未保存
```

**状态机**

| 状态名称 | 进入条件 | 退出条件 | 用户可见表现 |
|----------|----------|----------|--------------|
| 未反馈 | 当前歌曲未被操作 | 用户点击反馈按钮 | 喜欢按钮为空心 |
| 正反馈 | 用户点击喜欢 | 用户取消喜欢或切歌 | 喜欢按钮点亮 |
| 负反馈 | 用户点击不喜欢 | 切歌或撤销 | 显示“已减少类似推荐” |
| 已收藏 | 用户在 Programs 收藏节目 | 用户取消收藏 | 节目收藏按钮点亮 |
| 保存中 | 用户触发反馈 | 保存成功或失败 | 按钮短暂 loading |
| 保存失败 | 本地写入失败 | 用户重试或忽略 | 显示非阻塞提示 |

**字段规范**

| 字段名 | 类型 | 是否必填 | 长度限制 | 校验规则 | 错误提示 |
|--------|------|----------|----------|----------|----------|
| 事件类型 | 枚举 | 是 | 固定枚举 | `track_liked`、`track_like_removed`、`track_disliked`、`track_dislike_removed`、`program_favorited`、`program_favorite_removed`、`track_skipped` | “反馈类型无效” |
| 反馈对象 ID | 文本 | 是 | 128 个字符 | 必须对应当前歌曲或节目 | “当前内容无法反馈” |

**文案规范**

| 场景 | 文案内容 |
|------|----------|
| 空状态 | 播放后可以告诉 Koradio 喜不喜欢 |
| 加载中 | 正在记录你的反馈... |
| 操作成功 | 已记住你的偏好 |
| 操作失败 | 反馈保存失败，请重试 |

### 功能 6：服务配置与健康检查

**功能描述**

用户可在 Settings 页面管理设备级 `DeviceSettings`（数据目录和 Codex）以及当前档案的 `ProfilePreferences`（主题、DJ 语言和声音风格），并查看内置网易云 Provider 与 Apple 系统 TTS 的运行时健康快照。首次启动自动使用 OS 应用数据目录，不要求用户先选择路径。

**用户操作路径、界面元素、交互逻辑、数据变化**

- 用户打开 Koradio 但本地服务未响应 → 已打开或缓存的 PWA 显示 `○ OFFLINE`；只读 Settings 禁用全部配置、测试与迁移控件，仅显示本地服务启动说明、脱敏的上次状态和重试。未缓存的 PWA 不承诺离线可达。
- 用户点击 Settings → 系统分别读取共享 `DeviceSettings`、当前 `ProfilePreferences` 和运行时 `ServiceHealth` 快照。
- 用户填写 Codex 命令路径并点击“保存配置” → 普通配置写入 `DeviceSettings` → 用户看到“配置已保存”；内置网易云 Provider 和 Apple 系统 TTS 都不保存用户地址或密钥。
- 用户选择 `Theme Mode` 为 `Dark`、`Light` 或 `System` → 系统立即预览主题并保存偏好 → 用户看到 Radio 主播放页和详情页按对应昼夜视觉基准切换。
- 用户选择 `DJ Language` 为中文或英文 → 系统保存串讲语言偏好 → 用户下一次生成节目时听到对应语言的 DJ 串讲。
- 用户选择 `DJ Voice Style` 为 `British Soft Radio` → 系统将该声音风格传给 Codex 编排上下文，并在已安装的非 Personal Voice 中按 `en-GB` 与风格映射系统语音 → 用户听到最接近的英式电台主持人口吻；没有匹配语音时降级为文字 DJ。
- 用户点击“测试连接” → 系统分别检查 Codex、内置网易云 Provider 和 Apple 系统 TTS 原生 helper/标准语音能力 → 用户看到每项成功或失败；网易云和 Apple 系统 TTS 不出现凭据或 `NOT CONFIGURED`。
- 用户查看服务检测结果 → 系统汇总可用服务数量并展开失败项修复建议 → 用户看到 `3 OF 4 SERVICES AVAILABLE`、TTS 降级说明、返回 Radio 和修改配置按钮。
- 用户在 Settings 发起数据目录迁移 → 系统创建幂等异步迁移任务：验证目标为空且可写，暂停生成和播放，保存 checkpoint，备份并复制校验，原子切换 bootstrap 配置后重启服务；失败时回滚到旧目录，旧数据永不自动删除。
- 数据变化：设备配置写入 `DeviceSettings`，档案偏好写入 `ProfilePreferences`；网易云配置不持久化，`ServiceHealth` 仅是运行时快照。原始 Codex 输出和未脱敏诊断不得进入播放历史或普通日志。

**用户流程**

```text
[首次启动或 Settings 页面]
  ↓
[首次启动自动选择 OS 应用数据目录；用户查看服务状态]
  ├─ 本地服务未连接 → 显示 OFFLINE 异常页 → 重试或进入只读 Settings
  ├─ Codex 与网易云可用 → Radio 可生成节目
  │      ├─ Apple 系统 TTS 可用 → 生成语音 DJ
  │      └─ 原生 helper、匹配语音缺失或合成失败 → 文字 DJ 降级
  ├─ 用户切换 Theme Mode → 立即预览昼夜主题 → 保存偏好
  ├─ 用户切换 DJ Language → 下次生成节目使用对应语言
  ├─ 用户切换 DJ Voice Style → 下次 TTS 使用对应声音风格
  ├─ 用户发起数据目录迁移 → 异步执行阶段并发布进度 → 成功重启或失败回滚
  └─ Codex 配置缺失 → 用户填写配置并保存
         ├─ 保存成功 → 可测试连接
         └─ 保存失败 → 提示本地配置不可写
```

**状态机**

| 状态名称 | 进入条件 | 退出条件 | 用户可见表现 |
|----------|----------|----------|--------------|
| 未配置 | Codex 配置缺失 | 用户保存完整 Codex 配置 | Codex 配置显示未完成，网易云与 Apple 系统 TTS 单独显示能力状态 |
| 服务未连接 | 打开产品时本地服务未响应 | 重连成功 | 已缓存 PWA 显示 `○ OFFLINE`、只读 Settings、启动说明和脱敏诊断 |
| 已配置 | Codex 配置存在 | 用户修改 Codex 配置 | Codex 配置显示已填写，网易云与 Apple 系统 TTS 独立显示可用或降级 |
| 检测中 | 用户点击测试连接 | 检测完成或超时 | 每项显示检测中 |
| 检测结果 | 服务检测完成 | 用户返回 Radio 或修改配置 | 显示可用服务数量、失败项和修复建议 |

| 异常场景 | 触发条件 | 处理方式 | 用户反馈 |
|----------|----------|----------|----------|
| Codex 不可用 | 命令不存在或返回错误 | 阻止场景生成 | “Codex 未连接，请检查命令路径” |
| 内置网易云 Provider 不可用 | 健康检查失败或 15 秒超时 | 标记运行时不可用并允许重试，不展示地址或密钥表单 | “网易云服务暂不可用，无法搜索歌曲” |
| Apple 系统 TTS 不可用 | 原生 helper 缺失、标准语音不匹配、合成错误或 45 秒超时 | 允许文字 DJ 降级，不请求 Personal Voice 权限 | “系统语音不可用，将仅显示文字串讲” |
| 数据路径不可写 | 保存配置或历史失败 | 阻止相关写入 | “请选择可写的数据目录” |
| 数据目录迁移失败 | 目标验证、复制校验、原子切换或重启失败 | 回滚 bootstrap 配置并恢复旧目录；旧数据保留 | “迁移未完成，已继续使用原数据目录” |
| 完全离线 | Local Service 不可达 | 禁用配置、测试和迁移控件，只允许重试与查看启动说明/脱敏状态 | “本地服务未启动，设置暂时只读” |
| 主题保存失败 | themeMode 写入失败 | 回滚到上一次主题 | “主题保存失败，已恢复上一次设置” |
| DJ 语言无效 | 设置值不在枚举内 | 阻止保存并恢复默认中文 | “DJ 语言设置无效，已恢复中文” |

**多标签播放所有权**

- 同一设备与 Koradio origin 使用 `BroadcastChannel + localStorage TTL lease` 选出唯一主控标签；租约携带当前 `profileId`，主控每 `2s` 续约，租约 `5s` 未续约即过期。
- 被动标签只读展示低频状态，可申请接管，但不得启动第二条音频时间线或提交播放控制。
- 主动交接时，原主控先保存 checkpoint 并停止播放，再释放租约；异常过期后的新主控从最近 checkpoint 恢复。
- 所有命令携带 lease epoch；Local Service 和标签页均丢弃旧 epoch 的迟到控制事件。

### 功能 8：品味档案查看与编辑

**功能描述**

用户可在 Taste 页面查看由系统自动推导并与人工规则合并的个人音乐品味，使 Codex 后续推荐有明确且可控的偏好依据。

**用户操作路径、界面元素、交互逻辑、数据变化**

- 用户点击 Taste → 系统读取自动 `TasteProjection`、人工 `TasteOverrides` 和合并后的 `EffectiveTaste` → 用户看到系统推断、人工规则和最近反馈摘要。
- 用户点击“编辑标签” → 系统只编辑 `TasteOverrides` → 用户可新增、删除、排序人工标签与规则。
- 用户点击“保存品味” → 系统写入 `TasteOverrides` 并重新计算 `EffectiveTaste` → 用户看到“已更新你的音乐品味”。
- 自动投影可由完整 `FeedbackEvent` 流重建；重建、回放或算法升级不得覆盖人工 overrides，冲突时人工规则优先。
- 合并使用稳定、保序、忽略大小写的去重：人工 tags、避雷规则和场景规则完整保留在各自字段前部；自动 tags 和 signals 只填充 contract 剩余容量。
- 人工避雷规则与自动 tag 或 affinity 文本相同时移除对应自动值；自动 avoid signal 在人工避雷规则之后填充剩余容量。`EffectiveTaste` 是实时合并的只读结果，不单独持久化或提供写入口。

**用户流程**

```text
[Taste 页面]
  ↓
[用户查看品味档案]
  ├─ 有数据 → 展示标签/规则/反馈摘要
  ├─ 无数据 → 展示播放和反馈引导
  └─ 用户编辑 → 保存 TasteOverrides → 合并 EffectiveTaste
       ├─ 成功 → 后续场景生成使用新偏好
       └─ 失败 → 保留编辑内容并提示重试
```

| 编辑中 | 用户点击编辑 | 保存或取消 | 显示可编辑标签和规则 |
| 保存中 | 用户点击保存 | 保存成功或失败 | 保存按钮 loading |

## 五、接口与数据约束

### 1. 本地服务能力

| 能力 | 说明 | 最小输入 | 最小输出 | 失败降级 |
|------|------|----------|----------|----------|
| 档案管理 | 创建、读取、选择、更新本地档案 | profile 字段 | 当前 profile | 本地目录不可写时阻止保存 |
| 场景规划 | 调用本地 Codex 生成节目计划 | 场景文本、taste、history、时间 | 结构化 program plan | Codex 失败时保留输入并提示重试 |
| 音乐搜索 | 调用网易云 API 搜索歌曲和播放链接 | 关键词、候选风格 | 歌曲元数据、播放 URL | 无结果时换关键词重试 2 次 |
| TTS 生成 | 调用 bundled macOS native helper，以 `AVSpeechSynthesizer` 和已安装标准语音生成 DJ 音频 | DJ 文案、语言、声音风格 | 受控音频引用、时长、可选 marker；无 marker 时估算时间 | 失败时保留文字 DJ |
| 时间线构建 | 将节目转换为可播放判别联合并确定顺序 | program plan | `PlaybackTimelineItem[]` | 单曲失败时 Browser Audio Engine 跳到下一项 |
| 播放 checkpoint | 在状态边界保存可恢复的低频播放快照 | program、timeline item、position、`leaseEpoch` | `PlaybackCheckpoint` | 保存失败不影响当前 Browser Audio Engine 播放；旧 epoch 写入被拒绝 |
| 历史记录 | 保存节目、播放、反馈 | program、track、feedback | history record | 保存失败时提示但不中断播放 |
| 数据目录迁移 | 以幂等异步任务迁移设备数据根目录 | 目标目录、idempotency key | `jobId`、阶段事件 | 失败回滚，旧数据不自动删除 |
| WebSocket 推送 | 推送领域变化、任务阶段和低频 snapshot | 当前连接 | 统一 event envelope | 不推送高频播放进度；断开时前端显示未连接并尝试重连 |

### 2. 最小数据对象

| 对象 | 必需字段 | 用途 |
|------|----------|------|
| Profile | id、radioName、nickname、avatarRef、createdAt、updatedAt | 区分本地用户档案；`avatarRef` 必须由受控上传生成 |
| TasteProjection | profileId、tags、affinities、avoidSignals、sourceVersion、updatedAt | 由反馈与历史自动重建的品味投影 |
| TasteOverrides | profileId、tags、avoidRules、sceneRules、updatedAt | 人工编辑的规则，优先级高于自动投影 |
| EffectiveTaste | profileId、projectionVersion、overrideVersion、resolvedTaste | 提供给 Codex 的合并只读视图，不单独成为写入入口 |
| MusicTrack | id、source、sourceTrackId、title、artist、album、duration、lyricStatus | 保存稳定歌曲元数据；短期播放 URL 属于运行时解析结果 |
| program | id、profileId、scenarioText、title、status、trackIds、createdAt | 保存一次电台节目 |
| DjScriptSegment | id、programId、type、language、text、displayText、estimatedTiming、ttsAudioRef? | 保存开场、转场、结束语；文字降级仍保留在这里 |
| PlaybackTimelineItem | `dj`：id、segmentId、audioRef、duration；或 `track`：id、trackId、resolvedAudioRef、duration | 有音频的判别联合；文字 DJ 不创建伪 item |
| PlaybackCheckpoint | profileId、programId、timelineItemId、positionMs、volume、status、leaseEpoch、savedAt | 每个 Profile 只保留最新低频恢复快照；实时进度不持久化，旧 lease 不得覆盖新状态 |
| FeedbackEvent | id、profileId、targetId、type、idempotencyKey、createdAt | append-only 显式反馈事件，type 使用功能 5 的固定枚举 |
| DeviceSettings | dataRoot、codexCommand、updatedAt | 设备级共享配置；v1 的内置网易云 Provider 和 Apple 系统 TTS 不保存用户地址或密钥 |
| ProfilePreferences | profileId、themeMode、djLanguage、djVoiceStyle、updatedAt | Profile 级展示与 DJ 偏好 |
| ServiceHealth | service、status、checkedAt、redactedSummary | 运行时快照，不是配置或历史事实 |

### 3. 配置接口与异步事件

| 接口 | 语义 |
|------|------|
| `GET/PATCH /api/v1/device-settings` | 读取或更新设备级非敏感配置；v1 不接受网易云地址、Cookie 或密钥字段 |
| `GET/POST /api/v1/profiles` | 列出或创建本地档案；创建命令要求 `Idempotency-Key`，并原子创建默认 `TasteOverrides` 与 `ProfilePreferences` |
| `GET/PATCH /api/v1/profiles/:profileId` | 读取或更新显式指定的本地档案；v1 不提供删除接口 |
| `POST /api/v1/profile-avatars` | 接收单个 `multipart/form-data` 头像文件，校验真实图片类型、大小与受控文件策略后返回 `avatarRef` |
| `GET/PUT /api/v1/profiles/current` | 读取或切换当前 Profile；切换命令显式携带 `profileId`，并按取消旧生成、丢弃迟到事件、保存并停止旧播放的顺序协调 |
| `GET/PATCH /api/v1/profiles/:profileId/preferences` | 读取或更新指定 Profile 的主题与 DJ 偏好 |
| `GET/PATCH /api/v1/profiles/:profileId/taste` | 读取 projection/overrides/effective，或只更新人工 TasteOverrides |
| `POST /api/v1/profiles/:profileId/feedback-events` | 使用 `Idempotency-Key` 追加固定枚举 FeedbackEvent；成功后更新 projection 并发布 `feedback.persisted` |
| `GET /api/v1/profiles/:profileId/programs` | 分页读取指定 Profile 的 Program 历史摘要；详情字段按需加载 |
| `GET /api/v1/profiles/:profileId/programs/:programId` | 读取指定 Profile 拥有的 Program、DJ segments、歌曲元数据和判别式 timeline 详情 |
| `GET /api/v1/profiles/:profileId/playback` | 读取指定 Profile 最新的低频 Playback checkpoint 与对应 timeline item；无 checkpoint 返回未找到 |
| `PUT /api/v1/profiles/:profileId/playback/checkpoints` | 保存显式携带 `leaseEpoch` 的低频 checkpoint；校验 Program/timeline ownership、位置边界和完成边界 |
| `POST /api/v1/device-settings/data-root-migrations` | 以 idempotency key 创建数据目录迁移，立即返回 `202 { jobId }`；重复命令返回同一任务 |

迁移阶段通过既有 WebSocket event envelope 发布，包含 `eventId`、`eventType`、`version`、`occurredAt`、可选 `profileId`、`correlationId`、`sequence` 和 `payload`。迁移事件类型为 `data_root_migration.stage_changed`，payload 中包含 `jobId`、`stage`、`status` 与脱敏错误码；不得包含路径中的用户名、密钥或文件正文。

### 4. Codex 结构化输出要求

Codex 作为本地编排大脑时，应输出可解析 JSON。MVP 最小结构如下：

```json
{
  "programTitle": "Monday Night Exhale",
  "scenarioSummary": "夜晚写作，需要安静但不死板的 BGM",
  "djLanguage": "zh-CN",
  "djPersona": "british-soft-radio",
  "djScripts": [
    {
      "type": "intro",
      "language": "zh-CN",
      "text": "今晚适合慢一点，但不要睡着。我先给你挑几首有呼吸感的歌。",
      "displayText": "今晚适合慢一点，但不要睡着。我先给你挑几首有呼吸感的歌。",
      "estimatedTiming": true
    }
  ],
  "musicQueries": [
    {
      "keyword": "If Bread",
      "reason": "温柔、低刺激，适合夜间写作开场"
    }
  ],
  "playlistIntent": {
    "energy": "low-mid",
    "mood": "calm",
    "avoid": ["过强鼓点", "高亢人声"]
  }
}
```

Codex 输出必须遵守以下规则：

- 根据 `ProfilePreferences.djLanguage` 输出中文或英文串讲；中文串讲可保留少量 `ON AIR`、`Now Playing` 等电台状态词。
- 根据 `ProfilePreferences.djVoiceStyle = british-soft-radio` 控制文案：柔和、有磁性、专业克制、带轻微冷幽默。
- 每次节目至少输出 `intro`；仅在风格明显切换、用户追问、关键歌曲或结束时输出 `segue` 或 `outro`。
- 不输出每首歌固定口播，不编造无法确认的音乐背景；事实不确定时只描述歌曲氛围、听感和用户场景。
- `displayText` 是前端展示文本，`text` 是 TTS 输入文本；二者默认一致，但英文串讲可在 `displayText` 保留更短句式。
- TTS 无分句时间戳时，`estimatedTiming` 必须为 `true`，前端按文本长度进行近似高亮。

解析失败时，本地服务不得保存或回显原始正文。日志只记录稳定错误码、correlation ID、schema 失败摘要和脱敏诊断元数据；前端统一收到“Codex 没有完成节目规划，点击重试”，不可解析内容不得进入节目、历史或日志正文。
