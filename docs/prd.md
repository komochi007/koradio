# Koradio 产品需求文档

## 一、概述

**文档信息**

| 版本号 | 日期 | 作者 |
|--------|------|------|
| V 1.0.0 | 2026-07-06 | 用户 |

### 1. 需求背景

- **目标用户现状**：目标用户是有长期听歌习惯、收藏歌单和明确音乐品味的音乐重度用户。他们会根据工作、写作、通勤、夜晚放松等场景主动找歌，但现有音乐产品多以算法流、榜单、歌单广场为主，难以像私人 DJ 一样理解当下语境并完成策展式播放。
- **核心痛点**：用户想要“说一句场景需求，就得到一段有开场、有串讲、有歌单、有播放反馈的私人 AI 电台”，而不是自己反复搜索、切歌、拼歌单。
- **当前问题汇总**：

| 问题点 | 问题描述 | 当前影响 |
|--------|----------|----------|
| 场景理解弱 | 普通播放器只知道歌曲和歌单，不理解用户此刻在做什么、想要什么氛围 | 用户需要手动搜索、试错和切歌 |
| 推荐解释弱 | 算法推荐通常不给出有品味、有情境感的推荐理由 | 用户感觉像被动接收列表，而不是被私人 DJ 策展 |
| 电台体验断裂 | 播放器、歌词、AI 文案、语音串讲通常分散在不同产品或功能里 | 无法形成“开场-播放-转场-反馈”的完整电台闭环 |
| 个人品味沉淀弱 | 用户的歌单、反馈、场景偏好没有被整理成可读、可编辑的个人品味档案 | 产品难以越用越懂用户 |

### 2. 需求目标

- **业务目标**：
  - 在 2-3 周 MVP 内完成小范围内测版，验证“场景输入到电台播放”的核心体验。
  - 支持本地多档案使用，让不同内测用户能在同一台机器上拥有独立品味、历史和设置。
  - 建立可扩展的本地服务架构，为后续 24/7 自动电台、日程联动和更多音乐源预留空间。

- **成功标准**：

| 指标 | 当前状态 | 目标状态 |
|------|----------|----------|
| 场景点歌闭环 | 当前无产品实现 | 用户输入场景后，系统能生成 DJ 串讲、歌单队列并开始播放 |
| 首次配置完成率 | 当前无配置流程 | 用户可在 5 分钟内完成档案创建、API Key 配置和首次播放 |
| DJ 串讲可用性 | 当前无串讲 | 每次节目至少生成 1 段开场串讲，TTS 失败时保留文字串讲 |
| 推荐反馈沉淀 | 当前无反馈记录 | 喜欢、不喜欢、收藏和跳过会写入本地档案与播放历史 |
| 异常可恢复性 | 当前无错误处理 | Codex、网易云、TTS、歌词、播放失败均有明确提示和重试入口 |
| Radio 三态原型一致性 | 当前 04/05/06 出图存在尺寸和位置漂移 | 空节目、正在播放、生成中三态共用同一套 Radio 骨架，Logo、头像、时间区、DJ 状态栏、输入框和底部 Tab 尺寸一致 |

### 3. 需求范围

- **包含**：
  - 本地 Web/PWA 竖版电台界面。
  - Node.js 本地服务中枢。
  - 本地 Codex 编排：读取用户档案、生成节目计划、输出结构化播放指令。
  - 网易云 API 音乐搜索、歌曲信息、播放链接、歌词获取。
  - Fish Audio 兼容 TTS 服务：将 DJ 串讲合成为音频。
  - 本地多档案：用户档案、品味标签、播放历史、反馈、服务配置。
  - Radio 主播放页：顶部品牌区、时间/ON AIR、播放器、队列、DJ 状态栏、DJ 对话区、底部输入的单列固定顺序。
  - DJ 状态栏详情态：全屏沉浸节目界面、深色声波区、浅色节目面、AI 串讲词、歌词、歌曲进度和节目整体进度。
  - 昼夜模式：夜间和日间都按冷调高级电台风格完整设计。
  - AI DJ 人格与语言：英式电台主持人气质，支持中文或英文串讲。
  - Library、Taste、Programs/History、Settings 辅助页面。

- **不包含**：
  - 云同步、跨设备同步和云端账号系统。
  - 商业支付、订阅、会员权益。
  - 公开社区、歌单广场、多人同时收听。
  - 完整 24/7 自动电台和复杂日程自动编排。
  - 多音乐源聚合，第一版仅以网易云 API 为 MVP 音乐源。
  - 真实音频波形预分析，MVP 可使用前端模拟动态柱状图。

### 5. 产品体验与视觉原则

**体验气质**

Koradio 的第一版体验应是“私人电台陪伴感 + 简约高级风格”。产品需要像一个克制、懂音乐、不会过度打扰的私人 DJ，而不是工具型音乐控制台。界面保留点阵字体、`ON AIR`、状态灯、波形、队列、`LIVE` 状态等电台识别元素，但避免复杂装饰、过度拟物和大面积花哨渐变。

UI 品牌字标使用全大写 `KORADIO`，字距略微拉开；正文叙述和文档中仍使用产品名 Koradio。Logo 应采用单色几何低细节设计，避免直接使用麦克风、耳机、唱片等常见音乐应用符号。

**昼夜模式**

| 模式 | 视觉基准 | 使用规则 |
|------|----------|----------|
| Night | 深黑、墨蓝、低亮度边框、绿色状态光、白色点阵数字 | 默认首选模式，突出深夜私人电台感 |
| Day | 冷白、浅灰、黑色正文、绿色状态光、低饱和分割线 | 与 Night 使用同一布局和组件结构，不做简单反色 |

用户可在 Settings 中选择 `Dark`、`Light` 或 `System`。MVP 不根据歌曲封面自动变色；当前歌曲可影响小范围状态光或波形强弱，但不得改变整体主题基准。

**语言规则**

Radio 主播放页和 Radio Detail Sheet 尽量使用英文状态词，例如 `ON AIR`、`LIVE`、`QUEUE`、`PLAYING`、`SPEAKING`、`THINKING`、`Now Playing`、`Say something to the DJ...`。Onboarding/Profile、Library、Taste、Programs/History、Settings 以中文为主，降低配置和管理成本。DJ 串讲语言由 Settings 中的 `DJ Language` 决定，支持中文或英文，不做全应用完整国际化。

**AI DJ 人格与文案规则**

| 维度 | 规则 |
|------|------|
| 人格 | 英式电台主持人气质，声音柔和有磁性，专业克制，带轻微冷幽默 |
| 说话时机 | 每次节目必须有开场；只在情绪切换、风格变化、用户追问或关键歌曲前后做转场 |
| 说话长度 | 开场 20-45 秒；转场 8-20 秒；结束语 10-25 秒 |
| 内容范围 | 可简单介绍歌曲背景、风格特点、年代、场景关联或推荐理由 |
| 禁止内容 | 不油腻、不煽情过度、不装腔、不长篇百科、不每首歌都打断、不编造无法确认的音乐事实 |
| 失败降级 | 无法确认歌曲背景时，只描述可感知的氛围和用户场景，不输出虚构资料 |

**设计落地约束**

| 维度 | 规则 |
|------|------|
| 原型画布 | 高保真原型以 `960 × 1600px`、`3:5` 竖版桌面应用截图为基准，不展示设备模型、浏览器外框或营销落地页 |
| 内容列宽 | Radio/服务异常 `560px`；Profile/Onboarding `720px`；Library/Taste/Programs `800px`；Settings `820px`；Detail Sheet 覆盖完整 `960 × 1600px` 画布，浅色节目面内使用 `800px` 内容列 |
| 原型坐标 | Radio 04/05/06 使用固定生图骨架：Radio rail `x 200 / w 560`，输入框 `x 200 / y 1400 / w 560 / h 72`，Tab Bar `x 300 / y 1504 / w 360 / h 64` |
| 安全边距 | 原型页面左右和顶部使用 `56px` 安全边距；底部预留固定导航和安全区 |
| 底部导航 | 一级页面使用固定底部胶囊式 Tab Bar，`360 × 64px`，5 个 `72 × 64px` tab slot，选中底 `56 × 48px`；提供 Tooltip、`aria-label` 和键盘 Focus |
| 按钮颜色 | 主按钮使用黑白高对比，不使用绿色填充；绿色只用于 `ON AIR`、`LIVE`、`PLAYING`、`CONNECTED`、Focus Ring、细线和少量波形 |
| 字体 | 正文使用系统字体；等宽字体仅用于时间、数字、状态标签和队列序号；正文和 DJ 串讲不得使用点阵字体 |
| 页面结构 | Radio 始终保持中央窄列，不做桌面多栏；管理页保持单列，不做后台式侧边栏 |
| DJ 对话区 | Radio 主页面使用 `560 × 284px` 独立 dialogue well，保留弱背景和上下边界；DJ 文案用自然段，用户输入用右对齐弱气泡 |
| Detail Sheet | 覆盖完整产品界面，不露出底层 Radio 页面；使用深色声波区、纯白节目面 `#FFFFFF` 和浅灰串讲词/歌词卡片 `#F5F3F6`；不显示专辑封面、上一首、下一首、喜欢按钮或更多按钮 |
| 卡片使用 | 不把所有区块都做成悬浮卡片；Radio 中仅播放器、队列、输入区等需要边界的模块卡片化 |
| 错误提示 | 非阻断反馈可用 Toast；阻断错误必须在相关区块内联显示，并提供重试、跳转 Settings、修改输入或降级继续播放入口 |
| 动效 | 动效只表达声音、状态和空间关系；避免持续装饰动画、大幅弹跳、高频闪烁和横向页面滑动 |
| 无障碍 | 所有正式组件支持键盘操作、可见 Focus Ring、WCAG AA 对比度、`aria-live` 重要状态播报、200% 浏览器缩放和 `prefers-reduced-motion` |

## 二、页面结构

### 1. 导航结构

| 导航层级 | 页面名称 | 页面定位 | 入口 | 主要承载功能 |
|----------|----------|----------|------|--------------|
| 异常态 | 本地服务未连接 | 本地服务不可用时的恢复入口 | 打开产品且服务未响应 | 重连、前往 Settings、查看诊断信息 |
| 一级 | Onboarding/Profile | 首次进入、档案选择和档案创建 | 首次打开、顶部头像、登录/档案入口 | 创建本地档案、选择档案、基础偏好录入 |
| 一级 | Radio 主播放页 | 核心电台体验入口 | 默认打开、底部胶囊导航 Radio | 场景输入、DJ 串讲、播放控制、队列、反馈 |
| 一级 | Library | 音乐库和歌单来源管理 | 底部胶囊导航 Library、Radio 队列空状态入口 | 搜索歌曲、导入歌单、查看候选歌曲 |
| 一级 | Taste | 个人音乐品味档案 | 底部胶囊导航 Taste、Profile 完成后入口 | 查看和编辑风格标签、喜欢/避雷规则、场景偏好 |
| 一级 | Programs/History | 节目单与播放历史 | 底部胶囊导航 Programs、Radio 播放完成入口 | 查看历史节目、重播串讲、复用场景 |
| 一级 | Settings | 本地服务与密钥配置 | 底部胶囊导航 Settings、异常提示入口 | Codex、网易云 API、TTS、数据路径和连接状态 |
| 二级详情态 | Radio Detail Sheet | 当前节目沉浸详情 | 用户点击 Radio 页 DJ 状态栏 | 波形、AI 串讲词、歌词、歌曲进度、节目整体进度、单一播放/暂停 |

### 2. 页面职责说明

| 页面名称 | 用户目标 | 关键内容/组件 | 空状态 | 异常状态 |
|----------|----------|----------------|--------|----------|
| 本地服务未连接 | 恢复本地服务连接 | `○ OFFLINE`、断开波纹状态图标、重新连接、前往 Settings、`LOCAL SERVICE · NOT RESPONDING` | 本页面为空状态不适用 | 重连失败时保留诊断信息和 Settings 入口 |
| Onboarding/Profile | 创建或选择本地电台档案 | 头像、昵称、电台名、常听风格、默认场景、档案列表 | “还没有电台档案，创建你的私人电台” | 本地数据目录不可写时提示检查权限 |
| Radio 主播放页 | 输入场景并完成电台播放 | 单列固定顺序：顶部品牌区、时间/ON AIR、播放器、队列、DJ 状态栏、DJ 对话区、底部输入；主状态词使用英文 | `Say something to the DJ...` | 本地服务断开时显示“Koradio 服务未连接” |
| Library | 准备音乐候选来源 | 搜索框、歌单导入、歌曲列表、加入候选池按钮 | “还没有导入歌单，可先搜索一首歌试播” | 网易云 API 不可用时提示切换关键词或重试 |
| Taste | 管理个人音乐品味 | 品味标签、避雷规则、场景规则、反馈摘要、编辑按钮 | “播放和反馈后会在这里形成你的音乐品味” | 档案读取失败时提示重新选择档案 |
| Programs/History | 回看和复用历史节目 | 节目卡片、播放记录、DJ 文案、重播按钮、复用场景按钮 | “还没有节目，去 Radio 生成第一段电台” | 历史音频文件缺失时保留文本并提示无法重播音频 |
| Settings | 配置本地运行依赖 | 本地服务状态、Codex 路径、网易云 API、TTS Key、主题模式、DJ 语言、DJ 声音风格、缓存路径、测试按钮 | “完成必要配置后即可开始播放” | 密钥无效、服务超时、配置保存失败时显示具体原因 |
| Radio Detail Sheet | 沉浸查看当前节目 | 全屏沉浸节目界面、顶部状态、全宽动态波形、纯白节目面 `#FFFFFF`、节目标题、简洁当前歌曲、歌曲进度、`#F5F3F6` 串讲词/歌词卡片、节目整体进度、关闭按钮；DJ 段串讲为主，歌曲段歌词为主 | `No live session yet` | 歌词缺失、TTS 失败、播放失败时显示降级内容 |

### 3. Radio 主播放页布局规则

Radio 主播放页采用单列固定顺序，不因主题模式变化而改变信息架构。首屏优先建立电台仪式感，完整露出顶部品牌区、时间/日期、电台状态和播放器；队列、DJ 状态栏、DJ 对话区和底部输入从下方自然延续。空节目状态显示 `LIVE`，节目准备完成或正在播放时显示 `ON AIR`，生成中显示 `TUNING`。

高保真生图原型中的 04/05/06 三态必须共用同一套固定骨架：

| 固定区域 | 原型坐标与尺寸 | 说明 |
|----------|----------------|------|
| 顶部品牌工具区 | `x 56 / y 56 / w 848 / h 48` | Logo、`KORADIO` 字标、头像、主题按钮 |
| Radio 内容 rail | `x 200 / w 560` | 时间、播放器/空态/生成态、队列、DJ 状态栏、DJ 对话区、输入框 |
| 时间状态区 | `x 200 / y 136 / w 560 / h 132` | 时间、日期、`LIVE/ON AIR/TUNING/OFFLINE` |
| 状态主内容区 | `x 200 / y 320 / w 560 / h 220` | 空状态、播放器或生成中核心内容 |
| 队列区 | `x 200 / y 592 / w 560 / h 256` | 空队列、真实队列或骨架队列 |
| DJ 状态栏 | `x 200 / y 888 / w 560 / h 56` | 点击打开 Detail Sheet |
| DJ 对话区 | `x 200 / y 976 / w 560 / h 284` | 独立 dialogue well，DJ 自然段和用户弱气泡 |
| 底部输入 | `x 200 / y 1400 / w 560 / h 72` | 输入、语音和发送 |
| 底部导航 | `x 300 / y 1504 / w 360 / h 64` | 5 个等宽图标 Tab |

| 顺序 | 区块 | 组件细节 | 交互规则 |
|------|------|----------|----------|
| 1 | 顶部品牌区 | 头像、`KORADIO` 品牌字标、Profile 入口、`Dark/Light` 快捷切换 | 用户点击头像进入 Profile；点击主题切换即时预览并保存 |
| 2 | 时间/电台状态 | 大号点阵时间、星期、日期、状态点、`LIVE/ON AIR/TUNING/OFFLINE` | 状态随本地服务连接、节目生成和播放状态更新 |
| 3 | 播放器 | 当前歌曲、歌手、播放/暂停、上一首/下一首、停止、喜欢、音量、进度条 | 控制全局播放时间线，和 Detail Sheet 同步 |
| 4 | 队列 | `QUEUE`、曲目数、当前高亮曲目、前 3-5 首列表、`HIDE/LIST` | 空状态只显示状态栏；生成节目后默认展开 3-5 首 |
| 5 | DJ 状态栏 | `Koradio`、`LIVE/SPEAKING/THINKING/PLAYING`、状态灯 | 用户点击后打开全屏 Radio Detail Sheet |
| 6 | DJ 对话区 | `560 × 284px` dialogue well、DJ 自然段、用户弱气泡、时间、`REPLAY` | 展示开场和关键转场，不承担长篇聊天记录，不做聊天机器人界面 |
| 7 | 底部输入 | `Say something to the DJ...`、语音按钮、发送按钮、连接状态 | 用户发送场景后进入 `THINKING`，生成期间禁用重复发送 |

### 4. 页面状态矩阵

以下页面状态用于对齐 `design.md` 与高保真原型。实现时可复用路由和组件，但用户可见状态必须区分。

| 编号 | 页面/状态 | 内容宽度 | 必须出现的核心内容 |
|------|-----------|----------|--------------------|

### 功能 2：场景点歌电台生成

**功能描述**

用户可在 Radio 主播放页底部输入框描述当前场景，系统通过本地 Codex 编排生成节目计划、DJ 串讲、候选歌曲和播放队列，解决用户手动找歌和拼歌单的问题。

**用户操作路径、界面元素、交互逻辑、数据变化**

- 用户在输入框输入“今晚写东西，想要安静但不死板的 BGM” → 系统校验输入并锁定发送按钮 → 用户看到 DJ 状态栏变为 `THINKING`。
- 用户点击发送按钮 → 系统读取 taste、history、当前时间、`djLanguage`、`djVoiceStyle` 和服务配置并请求本地 Codex 输出节目计划 → 用户看到 DJ 对话区出现 `Tuning your station...`。
- Codex 生成串讲 → 系统按英式电台主持人规则控制语气、长度和说话频率 → 用户看到一段克制、有场景感的开场，而不是每首歌前都被打断。
- Codex 返回计划 → 系统调用网易云 API 搜索歌曲、获取播放链接和歌词 → 用户看到队列逐步生成。
- TTS 生成成功 → 系统把开场串讲音频插入播放时间线 → 用户听到 DJ 开场并看到串讲文本。
- 数据变化：新增 `program`、`dj_script_segments`、`play_queue`、`tts_audio`、`music_tracks`、`messages`，当前播放状态切换为 `playing_intro`，并记录本次使用的 `djLanguage` 与 `djVoiceStyle`。

**用户流程**

```text
[Radio 底部输入框]
  ↓
[用户输入场景并发送]
  ├─ 输入为空 → 提示输入场景，不发起请求
  └─ 输入有效 → 构建上下文
       ↓
     [请求本地 Codex 生成节目计划]
       ├─ 成功 → 搜索网易云歌曲
       │          ├─ 有歌曲 → 获取播放链接/歌词 → 生成 TTS → 创建队列 → 开始播放
       │          └─ 无歌曲 → 换关键词重试 2 次 → 仍失败则提示换个描述
       ├─ Codex 失败 → 保留用户输入并提示重试
       └─ 超时 60 秒 → 中断规划并提示重试
```

**状态机**

| 状态名称 | 进入条件 | 退出条件 | 用户可见表现 |
|----------|----------|----------|--------------|
| 空闲 | Radio 已连接且无生成任务 | 用户发送场景 | 输入框可编辑，DJ 状态为 `LIVE` |
| 校验失败 | 输入为空或超过限制 | 用户修改输入 | 输入框下方显示错误 |
| 规划中 | 用户发送有效场景 | Codex 返回、失败或超时 | DJ 状态为 `THINKING`，发送按钮禁用 |
| 搜歌中 | Codex 返回候选策略 | 搜歌成功或失败 | 队列区显示骨架项 |
| 合成中 | 歌曲候选可用且有 DJ 文案 | TTS 成功或失败 | DJ 状态保持 `THINKING`；TTS 段实际播放时 Detail Sheet 显示 `SPEAKING` |
| 可播放 | 队列和至少一首歌曲可用 | 播放开始 | 显示歌曲、串讲和队列 |
| 播放中 | 音频开始播放 | 暂停、结束或失败 | 播放按钮切换为暂停 |
| 失败 | Codex、搜歌或播放失败 | 用户重试或修改输入 | 显示错误提示和重试入口 |

**字段规范**

| 字段名 | 类型 | 是否必填 | 长度限制 | 校验规则 | 错误提示 |
|--------|------|----------|----------|----------|----------|
| 场景输入 | 文本 | 是 | 1-500 个字符 | 去除首尾空格后不可为空 | “告诉 DJ 你现在想听什么” |
| 语音输入转写文本 | 文本 | 否 | 500 个字符 | 转写失败时不覆盖原输入 | “没有听清，可以再说一次” |

**文案规范**

| 场景 | 文案内容 |
|------|----------|
| 空状态 | Say something to the DJ... |
| 加载中 | Tuning your station... |
| 操作成功 | Your session is on air |
| 操作失败 | 这次没有规划成功，请稍后重试 |
| 权限不足 | 请先在 Settings 配置 Codex 和网易云 API |
| 网络异常 | 音乐服务连接失败，请检查网络或稍后重试 |

**异常情况处理**

| 异常场景 | 触发条件 | 处理方式 | 用户反馈 |
|----------|----------|----------|----------|
| Codex 失败 | Codex 返回错误或非结构化结果 | 保留用户输入和上下文，允许重试 | “Codex 没有完成节目规划，点击重试” |
| Codex 超时 | 60 秒未返回 | 中断请求，释放输入框 | “规划时间太久了，请重试或缩短描述” |

| 未配置密钥 | 缺少 Codex/网易云/TTS 配置 | 阻止生成并跳转 Settings | “完成必要配置后即可开始电台” |

### 功能 3：播放控制与队列管理

**功能描述**

用户可在 Radio 主播放页控制当前节目播放，查看和管理队列，解决生成节目后无法主动调整播放节奏的问题。

**用户操作路径、界面元素、交互逻辑、数据变化**

- 用户点击播放/暂停按钮 → 系统暂停或恢复当前 TTS/歌曲音频 → 用户看到按钮状态、进度条和 DJ 状态同步变化。
- 用户点击下一首 → 系统结束当前音频并切换到队列下一项 → 用户看到当前歌曲、进度条、歌词和队列高亮更新。
- 系统生成节目成功 → 系统默认展开队列中的前 3-5 首歌曲 → 用户看到当前节目由哪些歌组成，并能确认当前高亮曲目。
- 用户点击 `HIDE/LIST` → 系统折叠或展开队列区 → 用户看到队列列表隐藏或显示。
- 用户拖动进度条 → 系统 seek 到目标时间 → 用户看到串讲高亮或歌词高亮同步跳转。
- 数据变化：更新 `playback_state`、`queue_current_index`、`queue_expanded`、`played_at`、`skip_events`、`listen_duration`。

**用户流程**

```text
[Radio 播放器区域]
  ↓
[用户操作播放控件]
  ├─ 播放/暂停 → 更新 playback_state → UI 同步状态
  ├─ 下一首/上一首 → 切换 queue index → 加载对应音频
  ├─ 拖动进度 → seek 音频 → 同步串讲/歌词高亮
  ├─ 生成后队列 → 默认展开 3-5 首
  ├─ 折叠队列 → 更新队列展示状态
  └─ 播放失败 → 尝试下一首或提示重试
```

**状态机**

| 状态名称 | 进入条件 | 退出条件 | 用户可见表现 |
|----------|----------|----------|--------------|
| 无队列 | 尚未生成节目 | 生成节目成功 | 只显示 `QUEUE` 状态栏和 `0 TRACKS` |
| 队列展开 | 节目生成成功 | 用户点击 `HIDE` | 默认展示前 3-5 首，当前曲目高亮 |
| 队列折叠 | 用户点击 `HIDE` | 用户点击 `LIST` | 只显示 `QUEUE` 状态栏和曲目数量 |
| 已就绪 | 队列已生成且音频可用 | 用户点击播放 | 显示当前歌曲和可播放按钮 |
| 播放中 | 音频播放开始 | 用户暂停、切歌、播放结束 | 按钮显示暂停，进度条推进 |
| 暂停 | 用户点击暂停 | 用户点击播放或切歌 | 按钮显示播放，进度停止 |
| 切换中 | 用户点击上一首/下一首 | 新音频加载成功或失败 | 当前歌曲区域显示加载状态 |
| 播放失败 | 音频加载或播放失败 | 自动跳过或用户重试 | 显示失败提示 |

**字段规范**

本功能无用户输入字段。

**文案规范**

| 场景 | 文案内容 |
|------|----------|
| 空状态 | 还没有播放队列，先告诉 DJ 你想听什么 |
| 加载中 | 正在加载下一段声音... |
| 操作成功 | 已切换播放 |
| 操作失败 | 当前音频无法播放 |
| 权限不足 | 当前档案不可写，播放记录不会保存 |
| 网络异常 | 播放链接连接失败，正在尝试下一首 |

**异常情况处理**

| 异常场景 | 触发条件 | 处理方式 | 用户反馈 |
|----------|----------|----------|----------|
| 播放失败 | HTMLAudio 报错或 URL 失效 | 自动跳过到下一首，可手动重试 | “当前音频无法播放，已尝试下一首” |

### 功能 6：服务配置与健康检查

**功能描述**

用户可在 Settings 页面配置本地 Codex、网易云 API、Fish Audio 兼容 TTS、本地数据路径、主题模式、DJ 串讲语言和声音风格，并检测服务是否可用，解决首次使用和异常恢复的问题。

**用户操作路径、界面元素、交互逻辑、数据变化**

- 用户点击 Settings → 系统读取当前配置和服务状态 → 用户看到 Codex、网易云、TTS、本地服务的连接状态。
- 用户填写 API Key 或服务地址并点击“保存配置” → 系统加密或本地保存配置引用 → 用户看到“配置已保存”。
- 用户选择 `Theme Mode` 为 `Dark`、`Light` 或 `System` → 系统立即预览主题并保存偏好 → 用户看到 Radio 主播放页和详情页按对应昼夜视觉基准切换。

  ├─ 配置完整 → 用户点击测试连接
  │      ├─ 全部成功 → 显示检测结果 → Radio 可生成节目
  │      └─ 部分失败 → 显示检测结果 → 标红失败项并给出修复入口
  ├─ 用户切换 Theme Mode → 立即预览昼夜主题 → 保存偏好
  ├─ 用户切换 DJ Language → 下次生成节目使用对应语言
  ├─ 用户切换 DJ Voice Style → 下次 TTS 使用对应声音风格
  └─ 配置缺失 → 用户填写配置并保存

|--------|------|----------|----------|----------|----------|
| 搜索关键词 | 文本 | 否 | 100 个字符 | 搜索时不可为空 | “请输入歌曲、歌手或专辑名” |
| 网易云歌单链接 | URL/文本 | 否 | 300 个字符 | 必须能解析出歌单 ID | “请输入有效的网易云歌单链接或 ID” |

**文案规范**

| 场景 | 文案内容 |
|------|----------|
| 空状态 | 还没有导入歌单，可先搜索一首歌试播 |
| 加载中 | 正在从网易云获取音乐... |
| 操作成功 | 已加入本地音乐库 |
| 操作失败 | 音乐获取失败，请重试 |
| 权限不足 | 请先在 Settings 配置网易云 API |
| 网络异常 | 网易云连接失败，请稍后重试 |

**异常情况处理**

| 异常场景 | 触发条件 | 处理方式 | 用户反馈 |
|----------|----------|----------|----------|
| 搜索无结果 | 网易云返回空数组 | 展示空状态和换词建议 | “没有找到相关歌曲，换个关键词试试” |
| 歌单解析失败 | 链接无合法 ID | 阻止导入 | “无法识别歌单链接” |
| 部分歌曲不可用 | 歌单内歌曲缺少播放链接 | 导入可用歌曲，标记不可用歌曲 | “已导入可用歌曲，部分歌曲暂不可播放” |
| API 限制或失败 | 网易云返回错误 | 保留输入，允许重试 | “网易云 API 暂不可用” |

### 功能 8：品味档案查看与编辑

**功能描述**

用户可在 Taste 页面查看和编辑由播放历史、反馈和导入歌单蒸馏出的个人音乐品味，使 Codex 后续推荐有明确的偏好依据。

**用户操作路径、界面元素、交互逻辑、数据变化**

- 用户点击 Taste → 系统读取本地品味档案 → 用户看到风格标签、喜欢歌手、避雷规则、场景规则和最近反馈摘要。
- 用户点击“编辑标签” → 系统打开标签编辑态 → 用户可新增、删除、排序标签。
- 用户点击“保存品味” → 系统写入 taste_profile 并刷新 Radio 上下文 → 用户看到“已更新你的音乐品味”。
- 数据变化：更新 `taste_profile.tags`、`avoid_rules`、`scene_rules`、`updated_at`。

**用户流程**

```text
[Taste 页面]
  ↓
[用户查看品味档案]
  ├─ 有数据 → 展示标签/规则/反馈摘要
  ├─ 无数据 → 展示播放和反馈引导
  └─ 用户编辑 → 保存 taste_profile
       ├─ 成功 → 后续场景生成使用新偏好
       └─ 失败 → 保留编辑内容并提示重试
```

**状态机**

| 状态名称 | 进入条件 | 退出条件 | 用户可见表现 |
|----------|----------|----------|--------------|
| 未形成 | 无播放历史和反馈 | 用户播放并反馈 | 显示引导文案 |
| 已形成 | 存在标签或反馈摘要 | 用户进入编辑 | 显示品味卡片 |
| 编辑中 | 用户点击编辑 | 保存或取消 | 显示可编辑标签和规则 |
| 保存中 | 用户点击保存 | 保存成功或失败 | 保存按钮 loading |
| 已更新 | 保存成功 | 用户继续编辑或离开 | 显示成功提示 |
| 保存失败 | 写入失败 | 用户重试或取消 | 显示错误提示 |

**字段规范**

| 字段名 | 类型 | 是否必填 | 长度限制 | 校验规则 | 错误提示 |

**文案规范**

|------|----------|

| 加载中 | 正在读取你的品味档案... |
| 操作成功 | 已更新你的音乐品味 |
| 操作失败 | 品味保存失败，请重试 |

|----------|----------|----------|----------|

### 功能 9：节目历史与复用

**功能描述**

用户可在 Programs/History 页面查看历史节目、重播 DJ 串讲、复用历史场景重新生成节目，解决好节目难以找回和复用的问题。

**用户操作路径、界面元素、交互逻辑、数据变化**

- 用户点击 Programs/History → 系统读取历史节目列表 → 用户看到节目标题、场景输入、歌曲数量、生成时间和收藏状态。
- 用户点击节目卡片 → 系统展示节目详情，包括 DJ 文案、歌曲队列、反馈结果 → 用户可查看当时的推荐理由。
- 用户点击“复用场景” → 系统把历史场景填入 Radio 输入框并可重新生成 → 用户看到 Radio 页进入待发送状态。
- 用户点击“重播串讲” → 系统播放已缓存 TTS 或回退到文字 → 用户听到或看到历史 DJ 串讲。
- 数据变化：读取 `program_history`，复用时新增一条 `messages` 草稿或新的 `program`。

**用户流程**

```text
[Programs/History 页面]
  ↓
[用户查看历史节目]
  ├─ 有历史 → 打开节目详情
  │      ├─ 重播串讲 → 播放缓存 TTS 或显示文字
  │      ├─ 复用场景 → 跳转 Radio 并填入历史场景
  │      └─ 收藏/取消收藏 → 更新 program_favorite
  └─ 无历史 → 引导前往 Radio
```

**状态机**

| 状态名称 | 进入条件 | 退出条件 | 用户可见表现 |
|----------|----------|----------|--------------|
| 无历史 | 没有 program 记录 | 生成第一个节目 | 显示空状态 |
| 列表 | 存在节目历史 | 用户打开详情 | 显示节目卡片 |
| 详情 | 用户点击节目 | 用户返回列表 | 显示 DJ 文案和队列 |
| 重播中 | 用户点击重播串讲 | 播放结束或失败 | 显示重播进度 |
| 复用中 | 用户点击复用场景 | 跳转 Radio | Radio 输入框填入历史场景 |
| 错误 | 历史读取失败 | 用户重试 | 显示错误提示 |

**字段规范**

本功能无用户输入字段。

**文案规范**

| 场景 | 文案内容 |
|------|----------|
| 空状态 | 还没有节目，去 Radio 生成第一段电台 |
| 加载中 | 正在读取节目历史... |
| 操作成功 | 已带着这个场景回到 Radio |
| 操作失败 | 节目历史读取失败 |
| 权限不足 | 当前档案不可读取 |
| 网络异常 | 历史保存在本地，网络异常不影响查看 |

**异常情况处理**

| 异常场景 | 触发条件 | 处理方式 | 用户反馈 |
|----------|----------|----------|----------|
| 历史为空 | 无 program 记录 | 展示空状态和 Radio 入口 | “去 Radio 生成第一段电台” |
| TTS 缓存缺失 | 历史音频文件被删除 | 显示文字串讲，不播放音频 | “串讲音频缺失，已显示文字版” |
| 历史读取失败 | 本地数据损坏 | 提供重试和回到 Radio | “节目历史暂时无法读取” |
| 复用失败 | Radio 状态不可用 | 保留在详情页并提示 | “Radio 未连接，暂时不能复用场景” |

## 五、接口与数据约束

### 1. 本地服务能力

| 能力 | 说明 | 最小输入 | 最小输出 | 失败降级 |
|------|------|----------|----------|----------|
| 档案管理 | 创建、读取、选择、更新本地档案 | profile 字段 | 当前 profile | 本地目录不可写时阻止保存 |
| 场景规划 | 调用本地 Codex 生成节目计划 | 场景文本、taste、history、时间 | 结构化 program plan | Codex 失败时保留输入并提示重试 |
| 音乐搜索 | 调用网易云 API 搜索歌曲和播放链接 | 关键词、候选风格 | 歌曲元数据、播放 URL | 无结果时换关键词重试 2 次 |
| TTS 生成 | 调用 Fish Audio 兼容服务生成 DJ 音频 | DJ 文案、声音配置 | 音频路径、时长、可选时间戳 | 失败时保留文字 DJ |
| 队列控制 | 管理播放队列和当前 index | program plan | play_queue、current item | 单曲失败时跳到下一首 |
| 当前播放状态 | 维护播放、暂停、seek、音量 | 控制动作 | playback_state | 状态保存失败不影响播放 |
| 历史记录 | 保存节目、播放、反馈 | program、track、feedback | history record | 保存失败时提示但不中断播放 |
| WebSocket 推送 | 推送播放状态、进度、队列变化 | 当前连接 | status event | 断开时前端显示未连接并尝试重连 |

### 2. 最小数据对象

| 对象 | 必需字段 | 用途 |
|------|----------|------|
| profile | id、radioName、nickname、avatar、createdAt、updatedAt | 区分本地用户档案 |
| taste_profile | profileId、tags、avoidRules、sceneRules、sourceSummary、updatedAt | 给 Codex 提供品味上下文 |
| music_track | id、source、sourceTrackId、title、artist、album、duration、playUrl、lyricStatus | 保存歌曲候选和播放信息 |
| program | id、profileId、scenarioText、title、status、trackIds、createdAt | 保存一次电台节目 |
| dj_script_segment | id、programId、type、language、text、displayText、startMs、endMs、estimatedTiming、ttsAudioPath | 保存开场、转场、结束语和降级高亮信息 |
| play_queue | programId、items、currentIndex、state、queueExpanded | 管理播放顺序和队列展示状态 |
| playback_state | profileId、programId、currentItemId、positionMs、volume、status | 同步播放器和详情卡片 |
| feedback_event | id、profileId、targetType、targetId、feedbackType、createdAt | 沉淀喜欢、不喜欢、收藏、跳过 |
| settings | profileId、codexCommand、neteaseApiUrl、ttsApiUrl、ttsKeyRef、dataPath、themeMode、djLanguage、djVoiceStyle | 保存本地服务、主题和 DJ 偏好配置 |

### 3. Codex 结构化输出要求

Codex 作为本地编排大脑时，应输出可解析 JSON。MVP 最小结构如下：

```json
{
  "programTitle": "Monday Night Exhale",
  "scenarioSummary": "夜晚写作，需要安静但不死板的 BGM",
  "djLanguage": "zh-CN",
  "djPersona": "british-soft-radio",
  "djScripts": [
    {
      "type": "intro",
      "language": "zh-CN",
      "text": "今晚适合慢一点，但不要睡着。我先给你挑几首有呼吸感的歌。",
      "displayText": "今晚适合慢一点，但不要睡着。我先给你挑几首有呼吸感的歌。",
      "estimatedTiming": true
    }
  ],
  "musicQueries": [
    {
      "keyword": "If Bread",
      "reason": "温柔、低刺激，适合夜间写作开场"
    }
  ],
  "playlistIntent": {
    "energy": "low-mid",
    "mood": "calm",
    "avoid": ["过强鼓点", "高亢人声"]
  }
}
```

Codex 输出必须遵守以下规则：

- 根据 `settings.djLanguage` 输出中文或英文串讲；中文串讲可保留少量 `ON AIR`、`Now Playing` 等电台状态词。
- 根据 `settings.djVoiceStyle = british-soft-radio` 控制文案：柔和、有磁性、专业克制、带轻微冷幽默。
- 每次节目至少输出 `intro`；仅在风格明显切换、用户追问、关键歌曲或结束时输出 `segue` 或 `outro`。
- 不输出每首歌固定口播，不编造无法确认的音乐背景；事实不确定时只描述歌曲氛围、听感和用户场景。
- `displayText` 是前端展示文本，`text` 是 TTS 输入文本；二者默认一致，但英文串讲可在 `displayText` 保留更短句式。
- TTS 无分句时间戳时，`estimatedTiming` 必须为 `true`，前端按文本长度进行近似高亮。

解析失败时，本地服务必须把原始输出保存到调试日志，并向前端返回“Codex 没有完成节目规划，点击重试”，不得把不可解析内容直接展示为节目。
