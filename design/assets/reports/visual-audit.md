## 0. 2026-07-13 尺度复核补正

01–06 HTML 完成后的 1:1 视觉复核证明，原始审计正确测得了参考图的大尺度，但后续裁决错误地把“小规格文档”置于参考图比例之上。由此产生的主要问题包括：Radio 内容列过窄、页面留白过大；Tab Bar、文字、表单、卡片、封面和头像整体偏小；顶部头像与右侧按钮天然不等大；页面 CSS 重复硬编码旧值，使 Tokens 不能完整控制共享几何。

本次复核确认 Fit 模式未显示缩放百分比只会增加判断误差，不是页面内部相对比例错误的根因。新的权威尺度为：

- 画布保持 `960 × 1600px`，页面安全边距保持 `56px`，顶部工具统一 `y 40 / h 64px`。
- Radio 共享 rail 为 `x 72 / w 816px`；时间、主内容、队列、DJ 状态、对话、输入和导航使用 `design/design.md` 的固定 slot。
- Profile、Management、Settings 分别使用 `848px`、`840px`、`832px` 单列；服务异常恢复区保留 `560px` 阅读宽度。
- Tab Bar 为 `620 × 88px`；顶部头像与图标按钮统一 `64 × 64px`；输入和大按钮为 `80px`。
- Profile 档案卡、档案头像、上传头像分别为 `256px`、`160px`、`144px`；Radio 当前封面、主播放按钮和状态点分别为 `132px`、`96px`、`12px`。

原始审计的 PNG 测量、产品语义检查、状态检查和视觉气质结论仍有效；涉及旧目标尺寸的 A-001、A-003、A-005、A-007、B-001–B-003 与 C-006 已由 [视觉差异与裁决记录](visual-decisions.md) 中的 2026-07-13 用户尺度校准取代。后续任务不得再引用本报告中的 `560px Radio`、`360 × 64px Tab`、`36px 顶部头像`、`64px Profile 头像` 等旧目标作为实现规格。

## 1. 仓库事实与边界

- 审计时仓库处于 Documentation-first 阶段，没有产品源码、依赖清单、运行脚本或可执行 HTML 视觉主源；后续 VDA-01–08 已建立当前 HTML 视觉主源范围。
- 审计时存在 15 张 Dark 高保真 PNG 参考图；当前 01–15 Dark HTML 已建立，其中 09 已通过自动检查与用户视觉验收，10–11 已通过自动浏览器验收与用户视觉反馈校准，12–15 已通过自动浏览器视觉、结构和文件验收；Light、响应式、完整交互与无障碍、正式截图基线和 Figma 派生物仍由后续任务完成。
- 本次只审计现有 PNG，不修改原图，不创建 `apps/`、`packages/design-tokens/`、包管理器或生产配置。
- PNG 是视觉参考，不是开发前视觉资产主源；`design/assets/prototype/` 中的 HTML / CSS / JavaScript 已是唯一主源，当前完整 Dark 页面范围为 01–15，整体视觉冻结仍依赖后续任务。

## 2. 审计方法

1. 校验 15 张源文件的存在性、格式、像素尺寸和 SHA-256。
2. 将 `971 × 1619px` 原图临时缩放为 `960 × 1600px`；缩放文件仅位于 `/tmp/koradio-vda00`，没有写入基线目录。
3. 对 Radio 04–06、Detail 07–08 使用半透明叠图，并叠加 `design/design.md` 的固定锚点。
4. 对一级页面底部导航进行同尺度裁切比较；对 Profile 与 Programs 主卡片进行比例对照。
5. 按产品语义、用户流程、明确视觉规范、PNG 未覆盖细节的顺序裁决。

归一化比例为：横向 `960 / 971 ≈ 0.9887`，纵向 `1600 / 1619 ≈ 0.9883`。整体导出缩放不计为差异；下文坐标均指归一化后的画布，约数使用 `≈`。

## 8. 限制与风险

- 静态 PNG 无法验证真实命中区、键盘路径、`aria-live`、WCAG 对比度、Reduce Motion 或交互状态。
- 当前 01–15 已可进行 DOM、CSS computed style 和局部自动视觉回归；全量 Light / 响应式与正式视觉回归仍未建立。
- PNG 中的 `http://localhost:4173` 只能视为参考图内的静态诊断 fixture；仓库没有已确认端口，不得将其描述为当前运行事实。
- 参考图含生成式图像漂移；除明确锚点外，不应反推每个像素为新规范。
